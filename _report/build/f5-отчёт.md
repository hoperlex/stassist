## 2. Что реализовано

### 2.1 Миграция и модель данных

`drizzle/migrations/0005_horoscopes.sql` (офлайн, `drizzle-kit generate`):
- `horoscopes` — по доку 22 §5: `scope`/`period`/`topic`/`status` enum'ы, `date_key text`,
  `body_md`, `humor bool`, `astro_events jsonb`, uniq `(scope,sign,period,topic,date_key,humor)`.
- `ai_report_kind` расширен значением `'personal_horoscope'` — **намеренно не добавлен** в
  `packages/shared` `reportKindSchema`/общий контракт `/ai-reports` (см. doc-комментарий
  `packages/db/src/schema/enums.ts`): у персонального гороскопа отдельный роут/контракт, `ai_reports`
  переиспользован только как хранилище с собственным репозиторием
  (`apps/api/src/repositories/personal-horoscope-repository.ts`), запросы общего репозитория
  явно исключают `kind='personal_horoscope'` (`ne(...)`), поэтому изоляция гарантирована и на
  уровне SQL, не только типов.

Применена и проверена на локальном Postgres 16 (`stassist_test`), включая `ALTER TYPE ... ADD
VALUE` в той же миграции.

### 2.2 `packages/llm/src/horoscope/*` — генератор текста

- **`astro-events.ts`** — `HoroscopeAstroEvents` (единая плоская структура для day/tomorrow/
  week/month/year/eastern/lunar_day) + чистые билдеры (`buildDayAstroEvents`,
  `buildRangeAstroEvents`, `buildYearAstroEvents`, `buildEasternYearAstroEvents`,
  `buildLunarDayAstroEvents`). Небо дня/недели/месяца строится **из уже предрасчитанного**
  `astro_calendar` (Ф3) — Ф5 не пересчитывает его заново (находка [полнота]); единственное, что
  считается заново — «точные аспекты транзитных планет дня» (снимок на референсный час опорной
  локации, майорные аспекты ≤3°, тот же принцип упрощения MVP, что и у `compute-day.ts`).
- **`templates.ts` + `phrase-banks.ts`** — детерминированный генератор текста (тот же
  архитектурный прецедент, что `corpus/build-corpus.ts` Ф4: буквальный `StubLlmProvider.generate()`
  даёт не-доменный плейсхолдер, непригодный для 300+ текстов на весь портал — см. doc-комментарий
  `generate-batch.ts`). Длины: день/завтра 600–900, неделя/месяц/год 1500–2500 — проверено
  юнит-тестами **на всех 12 знаках × 5 темах**, включая заведомо «пустые» астрособытия (нет
  ретрограда/void/фазы) — это оказалось критично: первый прогон `data:horoscopes-backfill` против
  реального Postgres выявил тексты короче целевого диапазона именно в этом случае (не покрывалось
  первой версией юнит-тестов, использовавшей «богатую» фикстуру) — исправлено добавлением
  гарантированных (не завязанных на опциональные факты) предложений, тест расширен до 323
  прогонов, повторный backfill подтвердил длины в целевом диапазоне без исключений.
- **`build-prompt.ts` + `tone.ts`** — промт для **реального** LLM-провайдера (батч на 12
  знаков/животных за вызов, структурированный JSON-ответ). Используется, только если
  `llm.name !== 'stub'` (`generate-batch.ts` решает по этому же признаку, что и
  `apps/worker/src/worker.ts` для выбора цепочки провайдеров).
- **`generate-batch.ts`** — единственная точка входа для worker/api: `generateZodiacHoroscopeBatch`,
  `generateEasternHoroscopeBatch`, `generateLunarDayHoroscope`, `generateHumorZodiacBatch`,
  `generateHumorProfessionHoroscope`. Автомодерация — переиспользует `postprocess/
  forbidden-filters.ts` Ф4 (не продублирована); битый/неполный JSON от реального провайдера →
  весь батч в `status='moderation'` с причиной, не публикуется молча.
- **`personal.ts`** — персональный гороскоп: `computeTransitBodyPositions` (Солнце..Плутон на
  момент «сейчас», без домов), `computeTransitToNatalAspects` (переиспользует
  `detectSynastryAspects` из astro-core), `prioritizeTransitAspects` (сортировка по орбису,
  вес по значимости тела), `buildPersonalHoroscopeSummary` (бесплатная краткая версия — ВСЕГДА
  детерминирована, без LLM), `buildPersonalHoroscopeFull` (полная версия — идёт через
  `LlmProvider.generate()` + ретрив по `aspect:{natal}:{angle}:{transit}` из корпуса Ф4, тот же
  путь, что и `ai_reports`, см. §2.4).

### 2.3 `apps/worker/src/horoscope/*` — cron-пайплайн

Cron `HOROSCOPE_QUEUE`, 00:20 МСК (21:20 UTC) — на 10 минут **позже** `ASTRO_CALENDAR_CRON`
(00:10 МСК), чтобы гороскопы читали свежий `astro_calendar` того же прогона, без гонки.

Каденс (`jobs.ts` `runHoroscopePipeline`, решает находку [неоднозначность]):
- **день+завтра** — каждый прогон (12×5×2=120 текстов/сутки, идемпотентно после первого);
- **неделя** — только если MSK-день недели = понедельник (12×5=60);
- **месяц** — только если MSK-число = 1 (12×5=60);
- **год (западный+восточный)** — идемпотентный чек текущего года каждый прогон + следующего с
  октября (doc 23 §2 «публикация к октябрю»);
- **лунные дни** (30, evergreen) и **шуточный контур** — идемпотентный чек каждый прогон.

**День+неделя+месяц+год × 12 знаков × 5 тем = ровно 300** (доразрешение DoD «300 текстов»
находки [неоднозначность]) — подтверждено и юнит-тестом (`sitemap.test.ts`
`zodiacHoroscopeUrls` = 240 + `yearlyGoroskopUrls` западная часть = 60 → 300 URL), и интеграционным
прогоном backfill (см. §3).

Идемпотентность **останавливает вызов LLM, не только запись** (`countExistingSignsForKey` перед
батчем — иначе повторный cron дёргал бы LLM даже когда писать нечего); проверено интеграционным
тестом со счётчиком вызовов `generate()`.

`page_cache` (requirement 6): реализована **запись+инвалидация по тегу** `horoscope:{date_key}`
после каждого успешного батча (`page-cache.ts`, `invalidatePageCacheByTag`), покрыто интеграционным
тестом. **HTTP-уровневое короткое замыкание** SSR-рендера в `apps/web/server/index.ts` (отдача
готового HTML из `page_cache` вместо повторного рендера через vike) **не подключено** в этой фазе —
ADR-7 (док. 21 §10) прямо обосновывает, что для предрасчитанной программатики горячий HTML-кэш
не критичен на MVP (SSR читает `/api/v1/horoscopes/*` — быстрый индексный поиск, не LLM); это
чистая аддитивная доработка следующей итерации, не блокер.

Алерт готовности (requirement 7): `HOROSCOPE_READINESS_QUEUE`, 01:00 МСК (22:00 UTC),
`checkDailyHoroscopeReadiness` — считает `status='published'` для `scope='zodiac', period='day'`,
ожидает 60; при недоборе — `logger.error({alert:true, ...})` (machine-readable поле для внешней
алертной системы; интеграция с Sentry/GlitchTip — вне MVP-скоупа, как и остальная real-time
алертная инфраструктура портала, см. док. 21 §9).

### 2.4 Персональный гороскоп (кабинет)

`GET /api/v1/personal-horoscope?birthProfileId=&period=day|week` (авторизация обязательна).
**Хранилище и ключ кэша** (находка [полнота]): `ai_reports.kind='personal_horoscope'`,
`cache_key = sha256(birthProfileId, 'personal_horoscope', period, dateKey, coreVersion,
promptVersion)` (`buildPersonalHoroscopeCacheKey`) — `dateKey` = день (period='day') или
ISO-неделя (period='week') → суточная/недельная инвалидация автоматически (новый день/неделя →
новый ключ). Генерация лениво при первом заходе за день, дальше — кэш-хит без пересчёта.

Краткая версия — **всегда бесплатна**, не требует LLM (детерминированный текст по топ-1
транзиту). Полная версия — **пейвол-заглушка** (`PERSONAL_HOROSCOPE_FULL_ENABLED=false`, тот же
паттерн, что `PREMIUM_REPORTS_ENABLED` в `routes/ai-reports.ts` Ф4): пока флаг выключен, полный
текст НЕ генерируется вообще (не тратит LLM-токены на контент, который никто не увидит) —
`fullMd=null`, `unlocked=false`. UI — карточка на `/profiles/{id}` (`PersonalHoroscopeCard.tsx`).

### 2.6 Разрешение неоднозначностей URL (задокументировано в коде)

- **300 URL-комбинаций** = произведение 5 периодов × 5 тем, а НЕ буквальное прочтение
  сокращённой записи doc 23 §2 (см. `packages/shared/src/schemas/horoscope.ts` doc-комментарий,
  `resolvePeriodOrTopicSlug`). Проверено тестом на отсутствие коллизий по всем 25 путям.
- **Коллизия `/goroskop/{znak}/{slug}` vs `/goroskop/{yyyy}/{znak}`** (оба глубины 3) —
  разрешена через `+route.ts` guard-функции (vike route functions) с регулярным выражением
  первого сегмента (год vs слаг знака — множества не пересекаются), логика вынесена в чистую
  тестируемую функцию `apps/web/lib/goroskop-route.ts`.
- **Годовая страница получает ГОД ИЗ URL, а не «текущий»** — реальный баг, найденный и
  исправленный при реализации: без явного `?year=` query-параметра на API `/horoscopes/zodiac/
  :sign/:period/:topic` страница `/goroskop/2027/{znak}` показывала бы контент ТЕКУЩЕГО года.
  Исправлено: `apps/api/src/routes/horoscopes.ts` принимает `year` только для `period='year'` и
  строит синтетическое `now` внутри нужного года.
- **Сборка клиентского бандла и `node:crypto`** — `packages/shared`'s barrel-индекс (без
  `sideEffects:false`) тянет `pd-cipher.ts`/`node:crypto` в клиентский vite-бандл при импорте
  ЛЮБОГО значения (не типа) из `@stassist/shared`. Решено тем же приёмом, что уже применялся в
  Ф0-Ф4 (`pages/natalnaya-karta/+Page.tsx`, `pages/privacy-policy/+Page.tsx`): клиентские
  компоненты импортируют напрямую из подмодулей (`@stassist/shared/schemas/horoscope.js`), а не
  из barrel; серверные `+data.ts`-загрузчики (`goroskop-page-data.ts`) вынесены в отдельные
  файлы, не импортируемые клиентскими компонентами.
- **`+route.ts` и pnpm-резолюция** — vike компилирует `+route.ts` отдельным упрощённым
  загрузчиком на этапе сборки, который не резолвит транзитивные зависимости воркспейс-пакетов
  (`zod` внутри `@stassist/shared`) в этой pnpm-монорепе. `goroskop-route.ts` намеренно НЕ
  импортирует `@stassist/shared` вообще (дублирует список RU-слагов знаков локально).

### 2.7 Стартовый backfill — `pnpm data:horoscopes-backfill`

`tools/data-horoscopes-backfill.ts` — data-step (§4 конвенций, требует `DATABASE_URL`, НЕ часть
build/CI-гейта), вызывает те же job-функции worker'а напрямую (собранный `dist`), идемпотентно.
Типичный прогон (не октябрь): **120 (день+завтра) + 60 (неделя) + 60 (месяц) + 60 (год) + 12
(восточный) + 30 (лунные) + 15 (шуточный: 12+3) = 357 текстов**, из них ровно 300 — полная
матрица `scope='zodiac'`. Проверено на локальном Postgres: полный прогон → 357 новых строк,
повторный прогон → 0 новых (идемпотентность), содержимое в целевых диапазонах длины (см. §2.2).

## 4. Плейсхолдеры и допущения (честно, не выдумано)

- **Профессии шуточного контура** — 3 выбраны редакционно (разработчик/бухгалтер/менеджер,
  промт называет первые два явно, разрешает «2–3»), не факт из внешнего источника.
- **Год восточного календаря** — опорная дата 15 июня (середина года, вне окна неопределённости
  Личунь ~4 февраля, см. предупреждение в `@stassist/astro-core` bazi) — та же оговорка ТОЧНОСТИ
  Ба-цзы, что уже задокументирована в ACCURACY.md пакета astro-core (Ф1), не новое допущение Ф5.
- **Годовой западный гороскоп** не сканирует ингрессы медленных планет (Юпитер..Плутон) за год —
  задокументированное упрощение MVP (`astro-events.ts` doc-комментарий `buildYearAstroEvents`);
  текст опирается на архетипику знака + ретроградные тела на 1 января, если есть.
- Плейсхолдеров `{{ЗАПОЛНИТ ЗАКАЗЧИК}}` в этой фазе не добавлено (Ф5 не работает с юридическими
  текстами/реквизитами).

## 5. Итог для оркестратора (кратко)

- **Гейт**: `typecheck` PASS, `lint` PASS, `test:unit` PASS, `build` PASS.
- **Маршруты**: см. таблицу §2.5 (13 типов страниц, полная матрица 300 URL для зодиакальной
  части + 25 восточных + 30 лунных + 16 шуточных = 371 индексируемый URL на старте).
- **Непустота**: (а) ленивая генерация при заходе — `apps/api/src/horoscope/lazy-generate.ts`,
  батч на 12 знаков/животных за промах кэша; (б) `pnpm data:horoscopes-backfill` — стартовый
  прогон, **357 текстов** (300 — полная зодиакальная матрица + 57 сверх нормы). Оба пути
  проверены интеграционными тестами на реальном Postgres 16.
- **URL шуточного**: антигороскоп `/shutochnyj-goroskop/{znak}` (scope='zodiac', humor=true);
