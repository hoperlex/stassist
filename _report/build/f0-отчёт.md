## 2. Что реализовано

### 2.1 Монорепозиторий
- pnpm workspaces (`pnpm-workspace.yaml`) + turborepo (`turbo.json`, задачи typecheck/lint/
  test:unit/test:integration/test:e2e/build/dev). `pnpm -r <script>` — прямой путь гейта,
  turbo — для `pnpm dev/build` на уровне репозитория и будущего CI-кеша.
- `apps/{api,web,worker}`, `packages/{astro-core,numerology-core,llm,shared,ui}`.
  `astro-core`/`numerology-core`/`llm`/`ui` — пустые каркасы (index.ts + плейсхолдер-unit-тест),
  наполнятся в Ф1/Ф4/Ф3/Ф6 согласно плану.
- TypeScript strict везде: единый `tsconfig.base.json` (target ES2022, module ESNext,
  moduleResolution Bundler, `strict`, `noUncheckedIndexedAccess`, `noEmitOnError`), path-алиасы
  `@stassist/*` → `packages/*/src` (используются typecheck/vitest/tsx/vite одинаково через
  `vite-tsconfig-paths`, без сборки промежуточных dist на этапе typecheck/test).
- ESLint 9 flat config (`eslint.config.js`, единый на весь монорепозиторий) + Prettier
  (`.prettierrc.json`) — раздельно: `lint` не форматирует, `format`/`format:check` — Prettier.
- vitest: `vitest.workspace.ts` в корне агрегирует `vitest.config.ts`/`vitest.integration.config.ts`
  каждого пакета; в каждом package.json — свои `test:unit`/`test:integration` (последний —
  `describe.skipIf(!process.env.DATABASE_URL)` внутри тестов).

### 2.2 `packages/shared`
- `config.ts` — zod-схема окружения (`envSchema`) + `parseConfig`/`loadConfig`: деление на
  «всегда обязательные» (порты, `APP_URL`, `COOKIE_SECRET`, `CORS_ALLOWLIST`) и «обязательные
  для функции» (`DATABASE_URL`, `S3_*`, `SMTP_*`, `NOMINATIM_*`, `YOOKASSA_*`, `*_API_KEY`).
  В `NODE_ENV=production` отсутствие переменных для выбранной не-stub подсистемы —
  `ConfigError` (fail-fast). В `development`/`test` — подсистема в `degraded`-списке, процесс
  поднимается. 6 unit-тестов покрывают оба режима.
- Порты + стаб-реализации (§2 конвенций), все — детерминированные и не-Lorem:
  `ObjectStorage`/`MemoryObjectStorage` (опционально пишет копию в `_work/tmp/storage`),
  `Mailer`/`ConsoleMailer` (лог + `_work/tmp/mail`), `Geocoder`/`FixtureGeocoder` (справочник
  городов РФ — Москва/СПб/Новосибирск/Екатеринбург/Казань/Владивосток с реальными
  координатами и IANA tz), `PaymentProvider`/`FakePaymentProvider`, `LlmProvider`/
  `StubLlmProvider` (осмысленный текст с пометкой «черновик», детерминирован по хэшу запроса),
  `EmbeddingProvider`/`StubEmbeddingProvider` (вектор размерности 1024 — под будущий
  `interpretation_chunks.embedding vector(1024)`). `createPorts(config)` — единая фабрика,
  сейчас работает только с `*_PROVIDER=stub`/аналогами, при реальном драйвере явно бросает
  «реальный адаптер появится в поздней фазе» (чтобы не притворяться работающим).
- `schemas/common.ts`, `constants.ts` — минимальные сквозные zod-схемы (uuid, пагинация,
  конверт ошибки API, health/ready-ответы) и константы (`REQUEST_ID_HEADER` и т.п.), без
  бизнес-логики.

### 2.3 `apps/api` (Fastify)
- Плагины: `@fastify/helmet`, `@fastify/cors` (allowlist из `config.corsAllowlist`, origin-
  callback), `@fastify/rate-limit` (100/мин, `/healthz`/`/readyz` явно исключены из лимита —
  иначе частые пробы health-check ложно попадали бы под rate-limit, см. §5).
- `fastify-type-provider-zod` (`.withTypeProvider<ZodTypeProvider>()`), request-id через
  `genReqId` (берёт `x-request-id` из входящего запроса или генерирует uuid), прокидывается в
  ответ и во все логи.
- pino JSON-логи с redaction (`apps/api/src/logging.ts`): пути `req.headers.authorization`,
  `req.headers.cookie`, `res.headers["set-cookie"]`, `req.body.password`, `req.body.token`,
  `req.body.refreshToken`, `req.body.email`, `req.body.birthDate`, `req.body.birthTime`,
  `req.body.placeName`, `req.body.lat`, `req.body.lon`. Кастомный `serializers.req` явно кладёт
  заголовки в лог (Fastify по умолчанию их не логирует) — иначе redaction был бы мёртвым кодом.
  Проверено unit-тестом с перехватом потока pino (`CaptureStream`): Authorization в логе
  заменён на `[скрыто]`, исходный токен в лог не попадает.
- `GET /healthz` — 200 всегда (liveness, не зависит от БД/секретов). `GET /readyz` — пингует
  БД через ленивый `pg.Pool` (создаётся только если есть `DATABASE_URL`), 503 без БД/при её
  недоступности, 200 при живом соединении.
- Graceful shutdown: `SIGTERM`/`SIGINT` → `app.close()` (`forceCloseConnections: 'idle'` —
  закрывает неактивные keep-alive сокеты сразу, активные запросы дожидаются ответа) →
  `closeDb()` → `process.exit(0)`.
- Конфиг — `packages/shared/config` (degraded локально, fail-fast в проде).

### 2.4 `apps/web` (vike SSR внутри Fastify + React 18 + AntD5)
- **Без `vike-react`**: расширение официально требует `react`/`react-dom` ≥19, что несовместимо
  с обязательным по ТЗ React 18 — реализован «ванильный» рендер-конвейер Vike:
  `renderer/+onRenderHtml.tsx` (сервер) и `renderer/+onRenderClient.tsx` (клиент), явно типизи-
  рованные `OnRenderHtmlAsync`/`OnRenderClientAsync` из `vike/types`.
- SSR-извлечение стилей AntD5: `@ant-design/cssinjs` (`createCache` → `StyleProvider` вокруг
  `<Page/>` → `renderToString` → `extractStyle(cache, true)`), CSS кладётся в `<head>` **до**
  тела — без «мигания» нестилизованных компонентов на первом кадре. Подтверждено curl-ответом:
  инлайновый `<style>`-блок с `ant-typography`/`css-dev-only-do-not-override-*` присутствует в
  HTML первого ответа.
- Страницы: `pages/index/+Page.tsx` («Stassist — скоро», AntD `Typography`) и `pages/app/+Page.tsx`
  (AntD `Card`/`Space`/`Button`, `useState`-счётчик с `data-testid="hydration-counter"` для
  проверки гидратации). `pages/_error/+Page.tsx` — страница ошибок Vike (без неё несуществующий
  маршрут рендерился как 500, см. п. 4 «Находки»); теперь корректно 404.
- `server/index.ts` — Fastify: dev — Vite dev-сервер в middleware-режиме (`@fastify/middie`),
  prod — `@fastify/static` отдаёт `dist/client/assets/**` под префиксом `/assets/` (НЕ весь
  `dist/client` под `/*` — иначе конфликт wildcard-роутов с catch-all, см. п. 4), catch-all
  `app.all('*', …)` вызывает `renderPage()` из `vike/server`. Graceful shutdown — тот же паттерн,
  что и в api.
- `pnpm --filter @stassist/web build` = один `vite build` (vite 7 + плагин `vike()` сам строит
  ОБА окружения — client и ssr — за один вызов; `dist/client` + `dist/server`).

### 2.5 `apps/worker` (pg-boss, ленивая инициализация)
- `Worker.start()`: если `config.db.url` не задан — `status='degraded'`, лог
  `infra not configured: db`, **без импорта/подключения pg-boss** — процесс не падает и не
  висит. Если `DATABASE_URL` есть — `new PgBoss(url)` → `start()` → `createQueue('ping')` →
  `work('ping', …)` → `schedule('ping', '*/5 * * * *')` (каждые 5 минут, точное API pg-boss
  12.26.0 сверено по установленным `.d.ts`, а не по памяти — `createQueue` перед `work`/
  `schedule`, `work`-хендлер получает МАССИВ джобов).
- `apps/worker/src/index.ts` держит процесс живым в degraded-режиме через `setInterval`
  heartbeat-лог; `SIGTERM`/`SIGINT` → `worker.stop()` → чистый `process.exit(0)`.

### 2.6 БД (`drizzle/`)
- `drizzle.config.ts` — `dialect: 'postgresql'`, оффлайн `drizzle-kit generate` (без
  `dbCredentials`, без introspection/pull — команда не касается сети/БД).
- Миграция `0000_extensions.sql` (custom, вручную) — `CREATE EXTENSION IF NOT EXISTS
  pgcrypto/vector/pg_trgm`, с явным комментарием, что применение требует Postgres с pgvector
  (`pgvector/pgvector:pg15`, а не голый `postgres:15`).
- Миграция `0001_users_audit_log.sql` (сгенерирована из TS-схемы `drizzle/schema/`) — skeleton
  `users` (id/created_at/updated_at — минимум, чтобы FK был валиден; Ф2 расширит ALTER-ом) и
  `audit_log` (actor_id nullable FK → users ON DELETE SET NULL, action/entity/entity_id/
  payload/request_id/created_at). users создаётся **в этой же (первой) миграции Ф0** — так FK
  валиден сразу, без промежуточного «FK без constraint» (это и есть выбранное в §5 конвенций
  разрешение конфликта, отменяющее более раннюю альтернативную рекомендацию из
  `_work/build/findings/f0.md` пункта db-migration-dependency, которая писалась до того, как
  было решено вносить skeleton users в Ф0).
- `tools/db-migrate.ts` — отдельный скрипт применения (`drizzle-orm/node-postgres/migrator`),
  не часть build/CI-гейта; без `DATABASE_URL` завершается с понятным сообщением и кодом 1
  (проверено).

### 2.7 Секреты, CI, Docker, Caddy
- `.env.example` — все переменные (app-конфиг + docker-compose + деплой), стабы по умолчанию
  (`LLM_PROVIDER=stub`, `STORAGE=stub`, `MAILER=stub`, `GEOCODER=stub`, `PAYMENTS=stub`);
  `.env` в `.gitignore` (уже было). Плейсхолдеры внешних фактов — см. п. 5.
- `.github/workflows/ci.yml`: job `gate` (единственный обязательный — typecheck→lint→
  test:unit→build, без docker/БД) + `integration` (реальный Postgres-сервис контейнер GH
  Actions, `continue-on-error: true`, не блокирует гейт) + `docker-images` (matrix api/web/
  worker, только `main`) + `deploy-stage` (условный по `vars.STAGE_HOST`, плейсхолдер).
- `docker-compose.yml` — `postgres` (`pgvector/pgvector:pg15`, **не** голый `postgres:15`),
  `minio`, `api`/`web`/`worker` (сборка из соответствующих `Dockerfile`), `caddy`.
- `apps/{api,web,worker}/Dockerfile` — multi-stage, non-root (`addgroup/adduser -S`),
  `pnpm deploy --prod` для минимального рантайм-каталога конкретного workspace-пакета.
- `caddy/Caddyfile` — `/healthz`, `/readyz`, `/metrics`, `/api/*` → `api:3001`; всё остальное →
  `web:3000`; `tls internal` по умолчанию (без реального домена — самоподписанный сертификат;
  с `CADDY_DOMAIN=<реальный домен>` caddy сам получит Let's Encrypt).

## 3. Верификация — что подтверждено вживую (не только unit-тестами)

Локально без Docker и без реальной БД был поднят каждый процесс напрямую через `tsx`/`node`
и опрошен `curl`/`autocannon`/Playwright-Chromium — воспроизводимо, но не автоматизировано в
гейте (ручной прогон в рамках этой фазы, инструкции ниже позволяют повторить):

- `GET /healthz` → 200 (degraded, без БД); `GET /readyz` → 503 (без БД) — как и требует
  верификация промта.
- `curl` с `Authorization: Bearer <секрет>` → в перехваченном pino-логе секрет отсутствует,
  вместо него `[скрыто]` (redaction подтверждён и unit-тестом, и живым запросом).
- SIGTERM api-процессу под нагрузкой: собственный скрипт на `node:http` (keep-alive, 5
  параллельных запросов каждые 20мс) слал трафик, пока в середине окна процессу не ушёл
  SIGTERM. Результат: 245 чистых `200 OK`, 0 «оборванных» ответов (`torn=0`, ни одного
  `res.on('aborted')` и ни одного некорректного JSON-тела), 470 корректных `ECONNREFUSED`
  **после** полного завершения процесса (это ожидаемо и не является «обрывом» — процесс уже
  закрылся). Процесс завершился по `SIGTERM` чисто (`ps` не находит PID).
- Тот же SIGTERM-под-нагрузкой сценарий повторён для `apps/web` (прод-режим, статика +
  SSR-catch-all) — чистое завершение, без зависаний.
- `apps/worker` в degraded-режиме (без `DATABASE_URL`): процесс не падает, остаётся живым
  (heartbeat), `SIGTERM` → чистый `process.exit(0)`.
- `/` рендерится сервером: `curl http://localhost:PORT/` содержит полностью готовый HTML
  (`<h1>Stassist</h1>` + текст) без выполнения JS.
- `/app` гидрируется — подтверждено **не только curl'ом**, а реальным headless-браузером
  (Playwright + Chromium, установлен через `pnpm exec playwright install chromium --with-deps`,
  реально удалось в этом окружении): `apps/web/e2e/hydration.e2e.test.ts`, 2/2 теста зелёные
  под `RUN_E2E=1` — клик по кнопке после `waitUntil: 'networkidle'` дважды увеличивает
  `data-testid="hydration-counter"` (0→1→2), что возможно только если React реально навесил
  обработчик на серверный HTML (т.е. гидратация отработала).
- 404 на несуществующий маршрут — до добавления `pages/_error/+Page.tsx` Vike **без** страницы
  ошибок рендерил такие маршруты как 500 (не 404) — это было исправлено (см. п. 4).
- `pnpm build` производственный режим (`NODE_ENV=production`, `dist/client`+`dist/server`)
  проверен вживую: статика по `/assets/*`, SSR по всем остальным путям, 404 на несуществующий.

## 4. Существенные находки/исправления в процессе реализации (для прозрачности)

1. **Rate-limit ложно резал health-check-трафик.** Изначальный `app.all` без исключений отдавал
   429 на `/healthz`/`/readyz` при частых пробах — типичная причина ложных «unhealthy» у
   оркестраторов. Исправлено: `config: { rateLimit: false }` на обоих роутах.
2. **`@fastify/static` под `'/'` конфликтует с catch-all SSR-роутом** (`FST_ERR_DUPLICATED_ROUTE`
   на `'*'`) — оба регистрируют wildcard на один и тот же паттерн. Исправлено: статика отдаётся
   только под `/assets/*` (совпадает с тем, куда vite реально кладёт бандлы).
3. **`vike-react` несовместим с React 18** (peer `>=19`) — реализован ванильный рендер-конвейер
   Vike (`onRenderHtml`/`onRenderClient`) вместо расширения; это также дало полный контроль над
   SSR cssinjs-экстракцией AntD5, которую сложно было бы сделать поверх готового рендерера.
4. **Без страницы ошибок Vike несуществующий маршрут отдавал 500, а не 404** — добавлен
   `pages/_error/+Page.tsx`.
5. **`noEmitOnError` не был явно выставлен** — при первой (ещё не исправленной) конфигурации
   сборки `apps/api` с нарушением `rootDir` (см. п. 6 ниже) tsc, несмотря на ошибки типов, всё
   равно эмиттировал `.d.ts`/`.js` рядом с исходниками `packages/shared/src/**`. Заражённые
   файлы найдены (`find … -name "*.d.ts"` вне `dist/`) и удалены; `noEmitOnError: true` добавлен
   в `tsconfig.base.json`, чтобы такой класс бага больше не проходил тихо. Финальный гейт (см.
   п. 1) прогнан уже ПОСЛЕ исправления, с нуля (`rm -rf node_modules **/dist`).
6. **`rootDir` конфликт при сборке зависимых пакетов.** Раз `tsconfig-paths` в dev/typecheck
   резолвит `@stassist/*` на исходники (`packages/*/src`), тот же паттерн в `tsconfig.build.json`
   тянул чужие `.ts` в компиляцию с `rootDir: "src"` → `TS6059`. Исправлено: `tsconfig.build.json`
   каждого пакета обнуляет `paths` (`"paths": {}`), заставляя `build`-компиляцию резолвить
   workspace-зависимости штатно через `node_modules` (symlink → package.json `types`/`main` →
   уже собранный `dist` зависимости — гарантируется топологическим порядком `pnpm -r build`).
7. **Мой собственный E2E-тест сперва «нашёл» ложный баг гидратации** (клик не менял счётчик) —
   на деле тест кликал ДО того, как клиентский бандл успевал догрузиться и гидрироваться;
   исправлено ожиданием `waitUntil: 'networkidle'` перед взаимодействием.
8. **Скрипт-гейт `test:e2e` с `RUN_E2E` изначально имел перепутанную логику `&&`/`process.exit`**
   (печатал сообщение «пропускаю», но всё равно запускал Playwright) — переписан на `||` с
   инвертированными кодами возврата, поведение проверено в обеих ветках (с `RUN_E2E=1` и без).

## 5. Integration/e2e/docker — что помечено «требует ручного шага»

## 6. Плейсхолдеры `{{ЗАПОЛНИТ ЗАКАЗЧИК}}`

Согласно §8 конвенций (агент не фабрикует внешние факты), в `.env.example` оставлены как
явные плейсхолдеры:

- `YOOKASSA_SHOP_ID`, `YOOKASSA_SECRET_KEY` — реквизиты платёжного провайдера (Ф8).
- `NOMINATIM_USER_AGENT` (часть значения — контакт для User-Agent Nominatim, требование их
  политики использования).
- `CADDY_DOMAIN` — реальный домен для TLS на проде (сейчас `localhost` + `tls internal`).
- `STAGE_HOST`, `STAGE_DOMAIN` — хост/домен стейджа для CI-джобы `deploy-stage`.

## 7. Известные ограничения / что сознательно не сделано в Ф0

- Docker-образы не собраны и не запущены реально (нет Docker в среде сборки) — Dockerfile'ы
  написаны по документированному паттерну (`pnpm deploy --prod`, non-root, multi-stage), но
  требуют первого реального прогона на машине с Docker.
- `packages/{astro-core,numerology-core,llm,ui}` — пустые каркасы, как и предписано промтом; в
  `packages/shared` уже реализованы порты `LlmProvider`/`EmbeddingProvider` (а не в
  `packages/llm`) — по прямому указанию задачи этой фазы (единая точка портов+стабов в Ф0,
  реальные адаптеры/RAG-конвейер — в `packages/llm` начиная с Ф4).
- CI job `deploy-stage` — заглушка (echo) без реального механизма доставки: нет заданного
  стейдж-хоста/реестра образов (см. плейсхолдеры выше).
- Метрики `/metrics` (prom-client) упомянуты в caddy-конфиге как маршрут «на будущее», сам
  эндпоинт в api ещё не реализован (появится вместе с наблюдаемостью в поздних фазах).
