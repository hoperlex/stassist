## 2. Биллинг: планы/подписка/платежи/промокоды/вебхуки (req.1-2 промта Ф8)

### 2.1 PaymentProvider — реальный адаптер + стаб дефолт

- `FakePaymentProvider` (packages/shared/src/ports/payment-provider.ts) — дефолт локально
  (`PAYMENTS=stub`), детерминирован по `idempotencyKey`, теперь поддерживает
  `savePaymentMethod`/`paymentMethodId` (рекуррент) и `refundPayment`.
- `YookassaPaymentProvider` (packages/shared/src/ports/yookassa-payment-provider.ts) — реальный
  REST-адаптер ЮKassa v3 (Basic Auth, `Idempotence-Key`, разовые/рекуррентные платежи через
  `payment_method_id`, возвраты, каркас чеков 54-ФЗ через `receipt`). Не поднимается в
  unit-гейте (сеть) — выбирается через `PAYMENTS=yookassa` (`ports/factory.ts`
  `buildPaymentProvider`), unit-тест `factory.test.ts` проверяет только инстанцирование
  (`instanceof`), без реальных HTTP-вызовов.
- СБП — не отдельный код-путь: ЮKassa сама предлагает способ оплаты на экране `confirmation_url`
  (документировано, требует активации в личном кабинете мерчанта — launch-checklist п.2).

### 2.2 Модель данных (миграция `drizzle/migrations/0008_f8_billing_marketing.sql`)

9 новых таблиц: `plans`, `subscriptions`, `payments`, `webhook_events`, `promo_codes`,
`experiments`, `experiment_events`, `quiz_answers`, `email_optouts` — точно по doc 22 §4/§7б.
Сиды (`drizzle/seed/0011_plans_experiments.sql`, идемпотентно `ON CONFLICT DO UPDATE`): 3 плана
(free/premium_m/premium_y) с ценами и фичами из `PLAN_CATALOG`, эксперимент `paywall_v1` с
вариантами `['control','trial_first']`.

### 2.3 Идемпотентность вебхука — юнит-тест двойной/тройной доставки

`apps/api/src/billing/webhook-processor.ts` — идемпотентная обработка вебхука ЮKassa за
интерфейсами `WebhookEventStore`/`PaymentsStore`/`SubscriptionsStore` (закрывает находку
[billing-без-провайдера] f8.md буквально: «логика идемпотентности через
in-memory-репозиторий»). `webhook-processor.test.ts` — **7 тестов БЕЗ БД/сети**, в т.ч.:
- тройная доставка одного `eventId` → эффект (`payments.updateStatus`/`subscriptions.updateStatus`)
  применяется **ровно 1 раз**, попыток вставки — 3 (проверено счётчиками на in-memory-стабах);
- `payment.succeeded` → подписка переходит в `active`; `payment.canceled` → `past_due`;
  `refund.succeeded` находит платёж по `object.payment_id` (не `object.id` — это ID возврата).

`eventId` — композитный ключ `${event}:${object.id}:${object.status}` (ЮKassa не отдаёт отдельное
поле «ID события», честно задокументировано в `webhook-processor.ts` и `routes/webhooks.ts`).

Подпись: `verifyWebhookSignature`/`signWebhookPayload` (HMAC-SHA256, `payment-provider.ts`) —
**наш собственный** дополнительный слой (задокументировано, что ЮKassa не выдаёт HMAC «из
коробки», реальная защита прода — IP allowlist, launch-checklist п.2), опционален через
`YOOKASSA_WEBHOOK_SECRET`; тест «подделка подписи → 401» — `payment-provider.test.ts` +
`routes/billing.test.ts` (живой HTTP через `app.inject()`).

Retry-очередь в worker (req.1 промта «retry-очередь в worker»): `apps/worker/src/billing/
webhook-retry-job.ts` подбирает `webhook_events.processed_at IS NULL` (эффект не применился
синхронно в API) и повторяет — переиспользует ИДЕНТИЧНУЮ `applyYookassaWebhookEffect`
(продублирована между `apps/api`/`apps/worker` намеренно, тот же паттерн, что весь остальной
проект — отдельные деплой-юниты без общего internal-пакета).

### 2.4 Подписка: триал/грейс/рекуррент/отмена

`POST /subscriptions` (apps/api/src/routes/subscriptions.ts): план + опциональный промокод →
подписка (`status='trial'` если `trialDays>0`) → `createPayment({savePaymentMethod:true})` →
`payments` строка. `POST /subscriptions/:id/cancel` — «отмена в один клик»: ставит
`cancelAtPeriodEnd=true`, статус НЕ меняется сразу (доступ до конца периода — юридическое
требование прозрачности, req.2). `apps/worker/src/billing/subscription-renewal-job.ts`
(ежедневный cron) — рекуррентное продление: `trial`/`active` с истёкшим `currentPeriodEnd` →
списание по `providerSubId` → `active`+новый период при успехе, иначе `past_due`; `past_due` за
пределами грейса (`SUBSCRIPTION_PAST_DUE_GRACE_DAYS=3`) → `expired`; `cancelAtPeriodEnd=true` +
истёкший период → `expired` напрямую. Статусы синхронизированы с doc 22 §4 (закрыто MINOR
[консистентность-модель-данных] f8.md).

### 2.5 Промокоды и минимальная админ-поверхность возвратов

`POST /promo-codes/validate` (публичный) — честные причины невалидности (`not_found`/`expired`/
`exhausted`, не просто `false`). `applyPromoCodeToPrice` (чистая функция, `billing.ts`) —
`percent_discount`/`fixed_discount_kop`/`trial_extension_days`.
`POST /payments/:id/refund` (роль `admin`) — закрывает MINOR [возвраты-из-админки] f8.md
минимальным способом (endpoint, без отдельной формы в кабинете — см. launch-checklist п.6).

## 4. Индивидуальные прогнозы / элективные окна (req.4 промта Ф8)

Закрывает MAJOR [полнота/зависимости] f8.md — алгоритм НЕ выдуман молча, задокументирован явно.

### 4.1 Алгоритм элективных окон (`packages/astro-core/src/electives/find-electives.ts`)

**Правила (дословно из doc-комментария файла, воспроизведены и на публичной `/methodology`):**
1. Интервал сэмплируется с шагом 12ч (standard) / 6ч (deep), максимум 120 суток (защита от
   неограниченного расчёта — каждая точка требует root-finding void-of-course по 9 телам).
2. В каждой точке — точные аспекты транзитных Солнце..Плутон+Луна к натальным (набор
   `major_minor`, стандартные орбисы проекта — `DEFAULT_ORB_BY_ASPECT`/`DEFAULT_ORB_BY_BODY`, НЕ
   отдельная эвристика).
3. Валентность: trine=+3, sextile=+2, quintile/biquintile=+1, conjunction=0 (нейтрально —
   природа соединения зависит от планет, сознательно не оценивается), semisextile/quincunx=-1,
   semisquare/sesquiquadrate=-1, square=-3, opposition=-3.
4. Вес пары тел — `BODY_SIGNIFICANCE` (намеренный дубль `BODY_WEIGHT` из персонального
   гороскопа Ф5, `packages/llm/src/horoscope/personal.ts` — astro-core не зависит от `@stassist/
   llm`).
5. Очко момента = Σ(валентность × (весНатал+весТранзит)).
6. Void-of-course Луна (`findMoonVoidOfCourse`, тот же движок, что `astro_calendar` Ф3) → штраф
   `VOID_MOON_PENALTY=-6`.
7. Ретроградный Меркурий — флаг всегда, штраф `MERCURY_RETRO_PENALTY=-4` только если вызывающий
   код передал `weighRetrogradeMercury=true` (вопросы о договорах/переговорах/поездках — не
   универсально для любого вопроса).
8-9. Точки группируются в окна (score > порога), окно = start/end/peakScore/avgScore/
   topAspects(≤3)/флаги; окна сортируются по `peakScore` убыв.

Покрыто **8 unit-тестами** (`find-electives.test.ts`): детерминизм (два прогона — идентичный
результат), сортировка/пороги, границы `MAX_SCAN_DAYS`/интервала, реальный найденный (не
угаданный по памяти) момент ретро-Меркурия для проверки штрафа.

### 4.2 Три конвейера заказа (req.4 «событие/период/вопрос»)

`customForecastSubjectSchema` (packages/shared/src/schemas/order.ts) — **`z.discriminatedUnion`**
по `type` (не единый объект с опциональными полями — так `Extract<…>` даёт настоящие узкие TS-типы
в apps/worker/src/forecast/):
- `event_date` → «транзиты на дату» — `scoreElectiveMoment` на конкретную дату, список аспектов
  + флаги void/ретро.
- `period_map` → «карта периода» (макс. 90 дней) — сэмплирование раз в 3 суток, топ-15 аспектов
  по точности орбиса за весь период.
- `electives_question` → «элективные окна в интервале» (макс. 120 дней) — прямой вызов
  `findElectiveWindows`, топ-5 окон + ключевые аспекты лучшего.

`apps/worker/src/forecast/generate-custom-forecast-job.ts` — оркестрация: заказ (`paid`) →
натальная карта из `charts` (уже посчитана при создании профиля, см.
`apps/api/src/charts/natal-chart-service.ts`) → расчёт по типу → короткое LLM-вступление (ТОЛЬКО
факты без ПД, конвейер Ф4-стиля) → PDF (переиспользован `renderPdfReport`/`report-content-types`
из Ф6) → `ai_reports`(kind='order') + `delivered` → письмо+уведомление. «Как считали» — раздел с
правилами прямо в каждом PDF (`CUSTOM_FORECAST_RULES_DISCLAIMER_RU`) — прозрачность видна
пользователю, не только в коде. Покрыто **4 unit-тестами** сборки контента (без I/O).

Цены — `CUSTOM_FORECAST_CATALOG` (3 типа × 2 глубины = 6 позиций, 390–1290 ₽, демо-значения как
и `PDF_PRODUCT_CATALOG` Ф6). UI — `/app/zakazat-prognoz` (мастер: тип → профиль → параметры по
типу → вопрос → глубина → оплата → поллинг статуса → PDF).

  прод (`SEO_NOINDEX_ALL=false` — launch-checklist п.4). Закрывает буквальную фразу промта
  «снятие noindex… — env-флаг».

- **Канонические URL** — уже была инфраструктура (`PageSeo.canonicalPath`), все 12 новых страниц
  её используют.
