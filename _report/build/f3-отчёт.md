## 2. Сверх гейта: живая верификация (не только unit-моки)

Поднят реальный стек против **живого Postgres 16 + pgvector** (`stassist_test`, локально в среде
сборки):

- Применены миграции 0000→**0003** (`pnpm db:migrate`), новые таблицы подтверждены `psql`
  (`\d compat_pages`, `\d astro_calendar`, `\d chart_shares`).
- Применён seed `0002_compat_pages_skeleton.sql` (`pnpm db:seed`) — **78 строк** в
  `compat_pages`, `body_md IS NULL` у всех (честно, Ф4 ещё не заливала тексты).
- `apps/api` (`node dist/index.js`) + `apps/web` (`tsx server/index.ts`, dev-режим с
  прокси `/api/*`) подняты одновременно, прогнаны реальные HTTP-запросы:
  - `POST /api/v1/calc/natal` — реальный расчёт `astro-core`, корректный `ChartData`.
  - `POST /api/v1/calc/synastry`, `POST /api/v1/calc/numerology/*` — работают.
  - `GET /api/v1/calc/compat-pages/telec/oven` → канонизировано в `{signA:"oven",
    signB:"telec", bodyMd:null}` (честный empty-state).
  - `GET /api/v1/calc/lunar-calendar/2026-08` → `{days:[], computed:false}` (worker ещё не
    предрасчитал — честно, не 500).
  - `GET /api/v1/calc/geocode/suggest?q=москва` (публичный, без авторизации) → подсказка «Москва».
  - `curl http://localhost:3010/sovmestimost/oven-i-telec` — **SSR-HTML содержит реальный
    текст без JS**: `<title>Совместимость Овен и Телец — гороскоп пары | Stassist</title>`,
    честный блок «текст пары… наполняется».
  - `curl http://localhost:3010/sovmestimost/telec-i-oven` → **301** → `/sovmestimost/oven-i-telec`
    (зеркальная пара). `curl .../oven-i-marsianin` → **302** → `/sovmestimost` (неизвестный знак).
  - **OG-шеринг сквозь весь путь**: `POST /api/v1/share` создал строку `chart_shares`
    (slug — sha256 от обезличенных `positions`); `apps/worker` `processPendingShares()` —
    прогнан вручную против той же БД — реально вызвал **resvg** и записал валидный
    **PNG 1200×1200** (`file` подтвердил `PNG image data, 1200 x 1200, 8-bit/color RGBA`) в
    `chart_shares.og_image_key`. `curl http://localhost:3010/podelitsya/<slug>` отдал SSR-страницу
    с реальным `<svg>` ChartWheel (те же данные, тот же рендер-код, что и в браузере).
  - Тестовые строки удалены из `stassist_test` после проверки (`DELETE FROM chart_shares`).

**Известное ограничение стаб-хранилища** (зафиксировано честно, не скрыто): `MemoryObjectStorage`
хранит объекты в памяти ПРОЦЕССА — worker и api в этом смоук-тесте были разными Node-процессами,
поэтому `GET /api/v1/share/:slug/og.png` из api-процесса не нашёл PNG, записанный worker-процессом
(вернул 404). Это ожидаемо для локального dev-стаба и НЕ баг Ф3: боевой `STORAGE=s3` (появится в
поздней фазе, см. `packages/shared/src/ports/factory.ts` — там уже зарезервирован драйвер, но
реальный адаптер сознательно не реализован раньше срока) даёт общее хранилище между процессами.
Логика на стороне API (читать по `og_image_key`, отдавать байты или честный 404) — корректна и
покрыта unit-тестами.

## 3. Что реализовано

### 3.1 `packages/ui` — ChartWheel и MatrixOctagram (изоморфные SVG)

Пакет был пустым каркасом (Ф0) — теперь полноценный: добавлены `react`/`@types/react`
(`peerDependency`), `@stassist/shared`, `@stassist/numerology-core`.

- **Ядро рендера — ЧИСТЫЕ функции без React**: `renderChartWheelSvg(input): string` и
  `renderMatrixOctagramSvg(input): string` строят полную SVG-строку напрямую (не JSX→строка).
  Это и есть «один код рендерит SVG и на SSR-страницах, и в PNG» (§6 док. 21): та же функция
  используется React-обёрткой (`ChartWheel.tsx`/`MatrixOctagram.tsx`, `dangerouslySetInnerHTML`)
  **и** worker'ом для resvg (без React вообще — см. `packages/ui/src/render.ts`, отдельная
  React-независимая точка входа `@stassist/ui/render`, чтобы worker не тянул React в граф
  зависимостей).
- **ChartWheel**: зодиакальное кольцо (12 знаков, глифы, подсветка по стихии), дома (куспиды +
  номера, ASC/MC/DSC/IC подписаны), планеты (глифы, ретро-маркер ℞), аспекты (цвет по типу:
  жёсткие/мягкие/нейтральные/минорные, минорные — пунктиром), светлая/тёмная тема, **биколесо
  синастрии** (карта B — во внутреннем кольце другим цветом, межкартовые аспекты — отдельным
  входом `crossAspects`). A11y: `role="img"`, `aria-label`, `<title>` на элементе и на каждом
  глифе (имя знака/планеты озвучивается скринридером).
- **MatrixOctagram**: классическая раскладка (прямой квадрат A/B/C/D + повёрнутый F1–F4 + центр
  E), диагонали F1–F3/F2–F4 подписаны как линия отношений/денежная линия. Рисует **только числа
  арканов** — трактовки не рисует (владелец текста — Ф4, см. §3.4).
- **Источник глифов** (закрыта находка «не оговорён источник»): только юникод-символы блока
  Miscellaneous Symbols (знаки зодиака, 10 классических тел) — шрифто-независимо, широкая
  поддержка и в браузере, и у resvg (fontdb ищет системные шрифты); для Хирона/узлов/Лилит/Селены
  — короткие кириллические аббревиатуры текстом (не экзотический астро-шрифт, чтобы resvg не
  «молча ломал» картинку при отсутствии эмбеда шрифта).
- **Unit-тесты** (требование промта «ChartWheel-рендер → валидная SVG-строка при заданном
  ChartData»): валидность SVG-обёртки, все 12 глифов знаков, планеты+ретро-маркер, дома/углы (и
  их отсутствие при `noHouses`), тёмная/светлая тема, биколесо+межкартовые аспекты,
  **детерминированность** (одинаковый вход → байт-в-байт одинаковый вывод — важно для resvg-кэша
  и для стабильности скриншот-тестов будущих фаз), экранирование XML в `title`.
- Известные упрощения (задокументированы в коде, не скрыты): без де-коллизии тесно стоящих
  глифов; Фортуна не отображается на колесе.

| `/kvadrat-pifagora` | дата → `POST /calc/numerology/psycho-matrix` → сетка 3×3 (традиционная раскладка 1-4-7/2-5-8/3-6-9) | «Текстовые значения ячеек психоматрицы наполняются» |
| `/chislo-puti` | дата → `POST /calc/numerology/life-path` → число + сумма цифр | «Краткое текстовое значение числа наполняется» |
| `/sovmestimost` | каталог всех 78 пар (перелинковка) + быстрый выбор двух знаков | — |
| `/sovmestimost/{a}-i-{b}` | SSR-текст пары из `compat_pages` (или empty-state) + виджет синастрии (2×форма → `POST /calc/synastry` → биколесо + межкартовые аспекты + «Поделиться») | «Текст разбора пары… наполняется» |
| `/lunnyj-kalendar` | «сегодня»-виджет из предрасчёта (или empty-state) + ссылка на текущий месяц | «Календарь наполняется» |
| `/lunnyj-kalendar/{yyyy-mm}` | таблица дней месяца из предрасчёта (или empty-state) + пред/след. месяц | «Календарь на этот месяц наполняется» |
| `/podelitsya/{slug}` | обезличенный снапшот `chart_shares` → ChartWheel + og:image (если готов) | «Картинка для соцсетей ещё готовится» |

Плюс `/sitemap.xml` (78 пар + калькуляторы + ближайшие 4 месяца лунного календаря) и
`/robots.txt` (запрет `/app`, `/api/`, `/profiles`) — зарегистрированы в `apps/web/server/index.ts`
до catch-all vike-роута.

**Дисклеймер** «информационно-развлекательный характер» (`lib/InfoDisclaimer.tsx`) — на всех
результато-производящих страницах (натал, матрица, психоматрица, число пути, пара совместимости,
лунный календарь).

**SEO-обвязка**: `apps/web/lib/seo.ts` (`PageSeo`) + правка `renderer/+onRenderHtml.tsx` — теперь
читает `pageContext.data.seo` (если страница отдаёт его через `+data.ts`) и подставляет
title/description/canonical/og:title/og:description/og:image/JSON-LD `BreadcrumbList`. Старые
страницы Ф0–Ф2 без `+data.ts` продолжают получать дефолт сайта — обратная совместимость не
нарушена (проверено тестами `pages/index`, `pages/app`).

**SSR-фетчинг данных**: страницы, которым нужны серверные данные (пара совместимости, лунный
календарь, шеринг), используют `+data.ts` хуки vike (`export const data: DataAsync<T>`), которые
на сервере ходят в `apps/api` HTTP-запросом (`lib/server-api.ts`, `loadConfig().apiUrl`) — `web`
никогда не подключается к БД напрямую, только через REST, как и раньше. Редиректы/404 — через
`vike/abort` (`redirect(url, 301|302)`, `render(404)`), встроенный catch-all в `server/index.ts`
их корректно прокидывает (проверено `curl -D -`, см. §2).

### 3.3 API: анонимные расчётные эндпоинты (`apps/api/src/routes/*`)

Новые роуты, все **без авторизации**, регистрируются в `app.ts`:

- `POST /api/v1/calc/natal`, `POST /api/v1/calc/synastry` — обёртка над `computeChart` +
  новая функция `detectSynastryAspects` (`packages/astro-core/src/aspects/synastry.ts`,
  переиспользует существующий `detectAspects`, отдаёт только межкартовые аспекты через
  префиксы `a:`/`b:`).
- `POST /api/v1/calc/numerology/life-path|psycho-matrix|matrix-of-destiny` — прямая обёртка над
  `@stassist/numerology-core` (пакет уже был полностью реализован до Ф3 — переиспользован как
  есть, не дублировался).
- `GET /api/v1/calc/lunar-calendar/:yyyyMm`, `GET /api/v1/calc/compat-pages/:signA/:signB` —
  публичное чтение предрасчитанных/seed-данных, **деградируют до 200 с честным пустым
  результатом** без БД (не 503) — страница обязана быть непустой или иметь честный empty-state
  (§6 конвенций), это применено и к API-контракту.
- `GET /api/v1/calc/geocode/suggest` — публичный дубль `/api/v1/geocode/suggest` без авторизации
  (закрывает находку [internal-completeness] «не оговорено геокодирование для анонимных форм»).
- `POST /api/v1/share`, `GET /api/v1/share/:slug`, `GET /api/v1/share/:slug/og.png` — см. §4.

**Приватность подтверждена**: `req.body` в принципе не входит в pino-serializer запроса
(`apps/api/src/logging.ts` — сериализатор `req` отдаёт только `method/url/headers`), поэтому дата
рождения из `/calc/*` физически не может попасть в лог; отдельный unit-тест
(`routes/calc.test.ts`) прогоняет реальный запрос через перехватывающий `pino`-стрим и проверяет
отсутствие значения даты в логе. Ни один `/calc/*`-роут не пишет в БД.

## 4. Механизм OG-шеринга (закрывает находку «нет постоянного URL/og:image»)

Проблема из `_work/build/findings/f3.md`: анонимный расчёт не сохраняется (требование «без
сохранения ПД»), а PNG «по кнопке» — у расшариваемой ссылки нет ни постоянного URL, ни
существующего og:image к моменту обхода краулером.

**Решение (реализовано и проверено живым запуском, см. §2):**

1. По кнопке «Поделиться» (`apps/web/lib/ShareButton.tsx`) фронт вызывает `POST /api/v1/share` с
   **обезличенным** срезом результата (`SharePositions` — только bodies/points/angles/houses/
   aspects/meta, БЕЗ `input`/даты/времени/места рождения — прямой аналог `posts.chart_id` из
   §7 модели данных).
2. Слаг — **детерминированный** `sha256(canonical JSON).slice(0,16)`
   (`apps/api/src/share/slug.ts`) — повторный шеринг ТОГО ЖЕ результата отдаёт **тот же URL**
   (дедупликация через `ON CONFLICT (slug) DO NOTHING`, без разрастания таблицы).
3. Постоянный URL — `/podelitsya/{slug}` (SSR-страница, ChartWheel + og:title/og:description
   всегда валидны сразу).
4. PNG генерируется **асинхронно worker'ом** (`apps/worker/src/share/process-pending-shares.ts`):
   pg-boss cron `chart-share-og-backfill` (по умолчанию раз в минуту — см. `SHARE_OG_CRON` в
   `worker.ts`) сканирует `chart_shares WHERE og_image_key IS NULL`, рендерит ту же SVG-строку
   (`renderChartWheelSvg`, `@stassist/ui/render` — БЕЗ React), кодирует в PNG **resvg**
   (`@resvg/resvg-js`, только в worker — не в api/web, как требует док. 21 §6), кладёт в
   `ObjectStorage` под ключом `og/{slug}.png`.
5. `GET /api/v1/share/:slug/og.png` отдаёт байты, если `og_image_key` уже проставлен, иначе — 404
   с понятным сообщением (краулер получит валидные og:title/og:description без картинки до
   следующего тика воркера — не идеально, но честно и не ломает рендеринг ссылки).

**Разделены** «время самой генерации PNG» (≤2 с — подтверждено: resvg рендерит 1200×1200 SVG
практически мгновенно, см. живой тест §2) и «задержка публикации» (до одного тика cron — в проде
рекомендуется короче интервал, напр. 15–20 с, сейчас 1 мин для экономии cron-слотов в MVP). Это
явно зафиксировано как формулировка выбранного компромисса, а не скрыто.

**Известное ограничение** — cross-process `MemoryObjectStorage` (см. §2, «требует ручного шага»
ниже).

## 5. Опорная локация лунного календаря (закрывает находку «не задана»)

**Москва** (55.7558° с.ш., 37.6173° в.д., `Europe/Moscow`) — единый источник правды
`CALENDAR_REFERENCE_LOCATION` в `packages/shared/src/constants.ts`, переиспользуется и worker'ом
(предрасчёт), и api (отдача в ответе), и web (текст в UI: «по Москва (Europe/Moscow)» на обеих
страницах календаря — явно показано пользователю). Персональная гео-локация лунных дней —
фича кабинета за пределами MVP-предрасчёта (не реализована в Ф3, не заявлена как готовая).

Упрощение MVP (задокументировано в коде `apps/worker/src/astro-calendar/compute-day.ts`): вместо
честного посуточного сканирования интервалов знака/void считается ОДИН «снимок» на 12:00 по
местному времени опорной локации на каждый календарный день; редкие дни с двумя сменами знака за
сутки отразят только состояние на полдень.

## 7. Требует ручного шага (integration/e2e — не блокирует гейт)

- **Скриншот-тесты ChartWheel** (промт: «3 эталонные карты, светлая/тёмная тема, пиксельная
  стабильность») — не входит в `test:unit` (нужен headless-браузер/Playwright с эталонными
  снапшотами); детерминированность SVG-строки (байт-в-байт одинаковый вывод при одинаковом входе)
  подтверждена unit-тестом — это необходимое, но не тождественное скриншот-тесту условие.
  Требует `RUN_E2E=1` + chromium, добавляется в следующей итерации Playwright-сьюта.
- **Lighthouse ≥90 perf/SEO на `/natalnaya-karta`** — требует живого прод-билда + chromium
  (`lighthouse-ci`), не входит в build-гейт; страница SSR, минимальный JS на первом кадре, но
  формальный прогон не выполнялся в этой сессии.
- **PNG-шеринг в реальном Telegram** («ручная проверка», прямо указано в промте) — механизм
  проверен end-to-end локально (см. §2, §4), но проверка реального превью в Telegram требует
  публичного HTTPS-домена — не выполнялась.
- **`STORAGE=s3` для продовой согласованности worker↔api** (см. известное ограничение §2/§4) —
  сейчас работает корректно в рамках одного процесса/при общем `MemoryObjectStorage`; для
  раздельных процессов (прод-топология) нужен реальный S3-адаптер — по договорённости
  (`packages/shared/src/ports/factory.ts`) появится в поздней фазе, это НЕ регрессия Ф3.
- **Применение миграции 0003 и seed 0002 к прод/стейдж БД** — `pnpm db:migrate && pnpm db:seed`
  (требует `DATABASE_URL`), не выполняется автоматически сборкой.
- **Регистрация cron `astro-calendar-precompute`/`chart-share-og-backfill` в live-режиме** —
  срабатывает только при заданном `DATABASE_URL` (см. §3 конвенций, ленивая инициализация
  worker'а) — первый реальный прогон предрасчёта на 18 месяцев вперёд не выполнялся в этой
  сессии (unit-тесты покрывают компонентную арифметику на окне до 30 дней, полное окно — секунды
  CPU, не блокер, но требует ручного `pnpm --filter @stassist/worker start` с `DATABASE_URL` для
  фактического наполнения `astro_calendar`, после чего `/lunnyj-kalendar` перестанет показывать
  empty-state).
- **Формат `pnpm format:check`** — не входит в обязательный гейт; в репозитории уже есть 150+
  файлов (включая пре-существующие, не тронутые в Ф3), не отформатированных по правилам prettier
  — не регрессия Ф3, чинить весь репозиторий сразу не входило в объём фазы.
