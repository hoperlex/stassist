## 2. Дополнительная верификация — реальный Postgres 16 (сверх юнит-гейта)

В этой сессии был доступен локальный Postgres 16 (с pgvector) — как и в Ф6, использовал его для
верификации сверх обязательного гейта:

1. Поднял чистую БД `stassist_f7_verify`, прогнал `pnpm db:migrate` (все 8 миграций, включая
   `0007_f7_wiki_community.sql` — 8 новых enum'ов, 7 новых таблиц: `wiki_article_revisions`,
   `posts`, `comments`, `reactions`, `friendships`, `reports_ugc`, `reputation`, + GIN pg_trgm
   индекс на `wiki_articles`) и `pnpm db:seed` (все сиды, включая новые `0008`/`0009`/`0010`) —
   **зелено**.
2. Счёт строк в живой БД: **189 вики-статей** по 10 разделам (12 signs, 14 planets, 12 houses, 7
   aspects, 6 schools, 22 arcana, 30 lunar_days, 27 nakshatras, 2 stones, 57 glossary);
   `interpretation_chunks` — **1478 из 1601** привязаны к `source_article_id` (непривязанные —
   `numerology:*`/`element:*`/`cross:*`, по правилу это ожидаемо, см. §3.2); **20 knamenитостей**,
   у КАЖДОЙ `verified:false`.
3. **31 integration-тест** (`apps/api/src/routes/community.integration.test.ts` — 4,
   `wiki-and-celebrities.integration.test.ts` — 6, плюс существующие Ф0-Ф6 — 21) прогнаны против
   этой БД через `app.inject()` — **все зелёные**. Ключевые сквозные проверки промта:
   - пост с картой → JSON `chart` НЕ содержит `1990-05-17`/`55.7558`/`37.6173`/`Europe/Moscow`/
     `Москва`, нет ключей `input`/`kind` → комментарий-разбор → «отметить полезным» → репутация
     выросла (0 → >0);
   - жалоба на пост → появляется в `GET /moderation/reports` → решение модератора убирает её из
     очереди; обычный пользователь получает 403 на `/moderation/*`;
   - премодерация: первые 3 поста новичка — `pending`, 4-й (без нарушений) — `approved`;
   - друзья: односторонний `chartShared` → синастрия `computed:false, reason:'not_mutually_shared'`;
     взаимный → `computed:true` + реальный список межкартовых аспектов;
   - редактор правит статью (роль `editor`) → версия +1, снимок предыдущей версии в
     `wiki_article_revisions`.
4. **Живой E2E-смоук через реально поднятые `apps/api` + `apps/web` (dev-режим, проксирование
   `/api/*`)** — сверх того, что обычно проверяется в отчётах фаз: curl по HTTP подтвердил
   рендер SSR HTML с правильными `<title>`/канониклами:
   - `/planety/mars-v-lve` → `<title>Марс в знаке Льва — значение | Зодиакум</title>`, текст
     чанка виден в meta description;
   - `/wiki/planets/mars` → `<title>Марс в астрологии | Зодиакум</title>`;
   - `/arkan/3` → `<title>Аркан 3 — Императрица (Матрица судьбы) | Зодиакум</title>`,
     `rel="canonical" href=".../wiki/arcana/arkan-3"` (алиас-редирект по каноничности, не дубль);
   - `/karta/albert-eynshteyn` → `<title>Натальная карта Альберт Эйнштейн | Зодиакум</title>`;
   - `/wiki/poisk?q=Венера+в+Скорпионе` → в HTML есть `href="/wiki/planets/venus"` — **поиск
     находит «Венера в Скорпионе»** (буквальный пункт «Верификация» промта Ф7), подтверждено
     через реальный HTTP-запрос к реально работающему стеку, не только unit-тестом;
   - `/sitemap.xml` — 972 URL, включая 240 «планета-в-X», 22 аркана, 189 вики-статей, 20
     знаменитостей.
   - **Найден и исправлен живым smoke-тестом реальный баг**: `wikiArticleListQuerySchema.limit`/
     `celebrityListQuerySchema.limit` изначально капались на 100/200, а sitemap-генератор
     (`apps/web/server/index.ts`) запрашивает `limit=500` для полного каталога — 400 Bad Request
     молча терял вики/karta-URL из sitemap (пустой список, без ошибки на странице). Исправлено
     (лимиты подняты до 1000), задокументировано в схемах.

Тестовая БД и временные процессы удалены после прогона — в репозитории не осталось следов (только
предполагаемые файлы фазы, см. `git status`).

## 3. Что реализовано

### 3.1 Вики (M7) — обязательный минимум MVP + полный набор разделов промта

**Наполнение** (`packages/llm/src/corpus/wiki-content.ts`, data-step `pnpm data:wiki` →
`tools/gen-wiki.ts` → `drizzle/seed/0008_wiki_articles.sql`): **189 статей**, все
`status='draft'` (правило непустоты §6 конвенций — показываются на SSR с бейджем «требует
редактуры», не скрываются):

| Раздел | Кол-во | Источник текста |
|---|---|---|
| signs | 12 (обязательный минимум) | переиспользует `signOverviewText()` из `corpus/templates.ts` (тот же текст, что чанк `sign:*:overview`) |
| planets | 14 (обязательный минимум) | `planetOverviewText()`, 10 классических + chiron/north_node/lilith/selena |
| houses | 12 (обязательный минимум) | `houseOverviewText()` |
| aspects | 7 (обязательный минимум) | `aspectOverviewText()` для 5 мажорных + квинконс + полусекстиль (решение по неоднозначности «7 из 11», задокументировано в коде) |
| arcana | 22 | агрегирует 6 представительных позиций матрицы (`arcanumPositionText`) в связный обзор на аркан |
| lunar_days | 30 | авторские краткие традиционные ключевые темы + честный дисклеймер («народная традиция, не единый стандарт») |
| nakshatras | 27 | честный плейсхолдер-каркас: только канонические санскритские названия (public domain), явная пометка «джйотиш — версия 1.x, детальные трактовки не выдуманы» |
| schools | 6 | авторские обзоры, факты сверены с `docs/research/01-школы-астрологии.md` (современная западная, кармический слой, джйотиш, авестийская Глобы, традиционная/эллинистическая, хорар/электив) |
| stones | 2 | хаб-статьи с перелинковкой на полный каталог `/kamni` (Ф6) |
| glossary | 57 | авторские краткие определения базовых терминов |

**Связывание `interpretation_chunks.source_article_id → wiki_articles`** (закрывает МАЖОР
[missing-step]): правило задокументировано и реализовано ДВАЖДЫ синхронно — как чистая TS-функция
`articleSlugForChunkKey()` (`packages/shared/src/schemas/wiki.ts`, 11 unit-тестов) и как
эквивалентный набор bulk SQL `UPDATE...FROM` (`drizzle/seed/0009_link_chunks_to_articles.sql`,
детерминированно, идемпотентно). Правило: `sign:*:overview`→signs, `planet:*:overview`→planets,
`house:*:overview`→houses, `planet_in_sign`/`planet_in_house`/`point_in_sign`/`point_in_house`→
статья ПЛАНЕТЫ (не знака — явное решение), `asc_in_sign`→signs, `aspect:*:overview` и
`aspect:{a}:{angle}:{b}`→aspects (по углу), `arcanum:{n}:*`→`arcana/arkan-{n}`; `numerology:*` и
прочее — намеренно НЕ линкуется (null, не выдумка). Живой прогон: 1478/1601 чанков привязаны.

**240 SSR-страниц** `/planety/{planeta}-v-{znak}` и `/planety/{planeta}-v-{n}-dome`: ОДНА пара
роутов (`pages/planety/@slug/+route.ts`+`+data.ts`+`+Page.tsx`) обслуживает все 240 комбинаций
через `apps/web/lib/planety-route.ts` (regex-разбор URL, БЕЗ импорта `@stassist/shared` — та же
осознанная причина, что `goroskop-route.ts`: упрощённый `+route.ts`-загрузчик vike не резолвит
транзитивные зависимости воркспейс-пакетов). RU-слаги: планеты — новый файл
`packages/shared/src/schemas/planet.ts` (10 классических, `ruSlug` транслит), знаки — новое поле
`slugPrepositional` в `zodiac.ts` (12 явных грамматических форм предложного падежа — НЕ
алгоритмическое склонение, см. doc-комментарий). Контент — переиспользует существующий
`GET /calc/interpretation` (Ф4), отдаёт draft+reviewed (правило непустоты). Живой прогон: все 240
URL в sitemap, SSR-рендер подтверждён curl'ом.

**`/arkan/{1..22}`** (закрывает MINOR [страница-без-владельца]): реализован как алиас на
`/wiki/arcana/arkan-{n}` — свой короткий URL, но `canonicalPath` указывает на каноническую
вики-страницу (избегаем дублей контента для SEO), решение задокументировано в коде.

**Поиск (pg_trgm)**: GIN-индекс `wiki_articles_search_trgm_idx` на `title || ' ' || body_md`
(миграция 0007; расширение `pg_trgm` уже было создано в Ф0). Репозиторий комбинирует ILIKE по
значимым словам запроса (`extractSignificantWords()` — чистая функция, юнит-тест) с сортировкой по
`similarity()`. Живой прогон подтвердил: «Венера в Скорпионе» → находит статью «Венера».

**Версии статей**: новая таблица `wiki_article_revisions` — при правке редактора
(`PATCH /wiki-articles/:slug`, роль editor/admin) снимок предыдущей версии сохраняется
атомарно в транзакции ПЕРЕД применением правки; `GET /wiki-articles/:slug/revisions` — журнал.
Прогнано на живой БД.

**SEO**: хлебные крошки + `BreadcrumbList` JSON-LD на всех новых страницах, `Article` JSON-LD на
статьях/аркан-алиасах/planet-in-X, честный `noindex` на empty-state (не найдено/API недоступен —
принцип из Ф3-Ф6). Sitemap-кластер (`apps/web/lib/sitemap.ts`): `wikiHubUrls`, `wikiSectionUrls`,
`wikiArticleUrls` (реальные слаги через REST, тот же паттерн `stoneUrls`), `planetInSignAndHouseUrls`
(240, чистая функция), `arkanUrls` (22), `celebrityUrls` (реальные слаги).

### 3.2 Celebrities (M8.4) — малый верифицируемый набор + механизм импорта, без выдумки

Закрывает МАЖОР [content-provenance]. **20 персон** (`tools/gen-celebrities.ts`, data-step
`pnpm data:celebrities`): исторические/общественные фигуры, ДАТА рождения которых —
энциклопедически неоспоримый факт (Эйнштейн, Моцарт, да Винчи, Ньютон, Шекспир, Твен, Пушкин,
Толстой, Достоевский, Кюри, Дарвин, Бетховен, Черчилль, Ганди, Фрида Кало, Тесла, Хокинг, Дисней,
Остин, Агата Кристи) — **сознательно БЕЗ ныне живущих политических фигур**. У КАЖДОЙ:
`verified: false`, `time: null`, `timeUnknown: true`, `lat/lon: null` (координаты места НЕ
выдуманы — city-level геокодинг тоже требует внешнего источника), `source` — текстовая атрибуция,
`note` — явная причина (там, где есть историческая неоднозначность: юлианский/григорианский
календарь у Ньютона/Пушкина/Толстого/Достоевского, неточная дата у Шекспира/Бетховена,
альтернативный год у Фриды Кало — не молчим об этом).

Остальные (до сотни персон промта) — **не выдуманы**: вместо этого `POST /api/v1/celebrities/import`
(роль admin, CSV по `celebrityCsvRowSchema`) + шаблон
`tools/data/celebrities-import-template.csv` с буквальным `{{ЗАПОЛНИТ ЗАКАЗЧИК}}` в строке-плейсхолдере
и примером формата. Прогнан живой upsert-путь юнит- и интеграционно (репозиторий).

Галерея `/karta/{celebrity}`: честно показывает `hasEnoughDataToCompute: false` (нет координат) —
**без выдуманной карты**, ровно по формулировке промта «работают на том, что есть, с честным
empty-state»; под карточкой — обсуждение (посты `kind='gallery'`, `celebritySlug`), пусто →
«Пока нет обсуждений… войдите, чтобы начать первым».

### 3.3 Анонимная копия карты (req.1 M8) — закрывает МАЖОР [data-model-gap]

Решение переиспользует существующую инфраструктуру, а не плодит новую таблицу: `posts.chart_id`
— FK на `charts.id` (НЕ отдельная таблица) — как и предусмотрел doc-комментарий
`packages/db/src/schema/charts.ts`, оставленный ещё в Ф3/Ф7-preview («`birthProfileId`
намеренно nullable… зарезервировано для будущей анонимной копии (Ф7)»). Публикация поста с картой
создаёт НОВУЮ строку `charts` с `birthProfileId=null` и `data=anonymizeChartData(chart.data)`.

**`anonymizeChartData()`** (`packages/shared/src/schemas/calc.ts`) — переиспользует ТОТ ЖЕ
`sharePositionsSchema`, что и `chart_shares` (Ф3): whitelist-сборка объекта из
bodies/points/angles/houses/aspects/meta{houseSystem,zodiac,noHouses}, БЕЗ `input`/`kind`. Важное
открытие при написании теста: `ChartData.input` буквально ЭХО-ит дату/время/место/tz рождения
обратно в вычисленную карту — анонимизация НЕ noop-операция, а обязательный шаг (задокументировано
в коде, закрывает MINOR [community-anon-unit]). **3 unit-теста без БД** (whitelist полей;
JSON-строка не содержит палящих значений даты/координат/tz; `input`/`kind` отсутствуют) +
**1 сквозной integration-тест** через реальную БД (see §2, п.3).

### 3.4 Коммьюнити (M8) — не «ядро», полный набор промта с модерацией

Реализовано ПОЛНОСТЬЮ (не урезано до ядра — промт допускал сокращение, но объём укладывался в
бюджет фазы):

- **Посты** (`posts`): 3 kind (chart_review_request/discussion/gallery), лента с фильтрами
  новые/без ответа/популярные (`GET /posts?sort=`), `GET /posts/mine`, soft-delete автором.
- **Комментарии** (`comments`): дерево ОГРАНИЧЕНО 2 уровнями (проверка на уровне приложения —
  Postgres self-FK не может декларативно ограничить глубину рекурсии без триггера, а единственная
  точка записи — этот роут, триггер избыточен), денормализованный `posts.commentsCount`.
- **Реакции** (`reactions`): полиморфная `(entity,entityId)`, «одна реакция на объект на
  пользователя» (toggle/replace), денормализованный `posts.likesCount` для kind='like'.
- **Репутация** (`reputation`): «автор поста отмечает разбор полезным» →
  `comments.markedUsefulAt` (одноразово, `WHERE marked_useful_at IS NULL`) →
  `USEFUL_COMMENT_REPUTATION_POINTS=5` → пороговые бейджи (`badgesForScore`, чистая функция).
- **Друзья и синастрия** (`friendships`): направленная модель с ДВУМЯ булевыми полями
  `sharedByUser`/`sharedByFriend` (закрывает MINOR [data-model-gap] «взаимность согласия») —
  виджет синастрии (`GET /friendships/:id/synastry`, переиспользует `detectSynastryAspects` из
  `@stassist/astro-core`, тот же алгоритм, что публичный калькулятор совместимости Ф3) доступен
  ТОЛЬКО когда обе стороны `true`; честный `computed:false` + `reason` вместо 403/500.
- **Модерация** (req.5): премодерация первых 3 одобренных постов новичка
  (`decideInitialPostModeration()`, чистая функция) + автоклассификатор
  (`packages/llm/src/moderation/ugc-classifier.ts`, детерминированный regex, ПЕРЕИСПОЛЬЗУЕТ
  `detectForbidden` из постобработки ИИ-отчётов Ф4 — единый словарь запрещённого контента,
  добавляет financial/insult) → очередь `GET /moderation/queue` (роль moderator/admin) →
  approve/reject → уведомление автору при отклонении. **Верификационный набор промта «20
  провокационных постов, ≥18 перехвачены» — ПРОЙДЕН** (юнит-тест с фиксированным набором из 5
  категорий: предсказания смерти/болезней, порча/привороты, медицинские директивы, финансовые
  директивы, оскорбления).
- **Жалобы** (`reports_ugc`): `POST /reports-ugc` (любой авторизованный) → очередь
  `GET /moderation/reports` → `resolved`/`dismissed`.
- **Правила сообщества**: `/pravila-soobshchestva` (публичная SSR-страница) — оригинальный
  текст, вдохновлённый общепринятыми принципами этики астрологов (НЕ дословный перевод
  OPA/ISAR — задокументировано, не требует плейсхолдеров §8, т.к. не юридический документ и не
  содержит фактов об операторе).
- **Уведомления**: `comment_reply` (ответ на пост/комментарий), `friend_request`, `moderation`
