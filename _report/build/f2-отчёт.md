## 2. Сверх гейта: живая верификация (не только моки)

Гейт выше не требует БД/сети, но в этой сессии была реально поднята **Postgres 16 +
pgvector** (локально в среде сборки), применены миграции 0000→0002, засеяны системные
calc_presets, и:

- **9/9 integration-тестов** (`pnpm -r test:integration`, DATABASE_URL) — зелёные, включая
  полный цикл reuse detection против реальной БД.
- **Живой HTTP-стек**: `apps/api` (`node dist/index.js`) + `apps/web` (`tsx server/index.ts`,
  dev-прокси `/api/*`) подняты одновременно, прогнан **Playwright e2e** (`RUN_E2E=1`):
  регистрация → чтение письма ConsoleMailer → verify-email → логин → создание профиля
  (403 `consent_required`) → экран согласия 152-ФЗ → повторное создание профиля с
  автодополнением места (`GEOCODER=stub`, «Москва») → редирект на карту → натальная карта
  реально посчитана `astro-core` и отображена (таблица планет, «Солнце» видно). Тест
  закоммичен: `apps/web/e2e/cabinet-flow.e2e.test.ts`.
- **Шифрование на месте подтверждено `psql`**: `birth_date_enc`/`place_name_enc`/`lat_enc` в
  БД не содержат исходных значений, формат `v1:<iv>:<tag>:<ciphertext>` (см.
  `birth-profiles.integration.test.ts`).
- **Удаление аккаунта подтверждено `psql`**: e-mail анонимизирован, `status='deleted'`,
  `display_name IS NULL`, запись `user.account_deleted` в `audit_log` есть, повторный логин —
  401.

## 3. Что реализовано

### 3.1 Миграции и модель данных (`packages/db` — новый пакет)
Схема БД перенесена из `drizzle/schema/` (Ф0) в `packages/db/src/schema/*.ts` — иначе
`apps/api` не мог типобезопасно импортировать её как обычную workspace-зависимость без
конфликта `rootDir` при `tsc -p tsconfig.build.json` (см. находку rootDir, `_report/build/
f0-отчёт.md` §4.6). `drizzle.config.ts` обновлён, `drizzle/migrations`/`drizzle/seed` остались
единым каталогом (§5 конвенций).

Миграция `0002_auth_profiles_consents.sql` (ALTER `users` + 9 новых таблиц, все из списка
требования): `refresh_tokens`, `password_resets`, `email_verifications` (находка
[schema-gap] — отдельная таблица по аналогии с `password_resets`), `consents`,
`birth_profiles`, `calc_presets` (+ колонка `code` для идемпотентного сида), `charts`,
`celebrities`, `geocode_cache`. `users` ALTER добавляет email/password_hash/role/status/tz/
locale/deleted_at и т.д. Применено и проверено на живой БД — 11 таблиц.

`email` — уникальность через простой `unique()` + app-level нормализация (`.trim().
toLowerCase()`), а не `citext`-расширение Postgres (документированное осознанное отступление
от буквальной формулировки модели данных — см. комментарий в `schema/users.ts`).

### 3.2 Auth (`apps/api/src/auth/*`, `routes/auth.ts`)
- Регистрация → `email_verifications` (hash, TTL 24ч) → письмо (Mailer-порт); email уже занят
  → единый ответ + письмо «у вас уже есть аккаунт» вместо верификации (anti-enumeration).
- Логин → argon2id (`hash-wasm`, WASM — без node-gyp) + dummy-hash constant-time сравнение при
  отсутствии пользователя (`auth/dummy-hash.ts`).
- Access JWT — EdDSA/Ed25519 (`jose`), TTL 900с, claims только `sub`/`role`/`jti`/`iat`/`exp`
  (без e-mail).
- Refresh — opaque токен (`auth/tokens.ts`), httpOnly cookie; **ротация + reuse detection** —
  чистая логика `auth/refresh-rotation.ts` над интерфейсом `RefreshTokenRepository`
  (`auth/refresh-repository.ts`, in-memory для unit + `PgRefreshTokenRepository` для рантайма).
  Reuse → `revokeFamily` + `audit_log` событие `auth.refresh_reuse_detected`.
- CSRF — double-submit + HMAC (`auth/csrf.ts`), применён к `/auth/refresh` И `/auth/logout`
  (находка [security-scope]: не только SSR-формы).
- Password reset (TTL 1ч, опциональный отзыв всех сессий).
- Rate-limit: 10/мин на большинстве auth-эндпоинтов, 30/мин на `/refresh` (общий лимит — 100/мин).

### 3.3 Профили рождения (`birth-profiles-repository.ts`, `routes/birth-profiles.ts`)
CRUD + AES-256-GCM шифрование 🔒-полей на границе репозитория
(`packages/shared/src/crypto/pd-cipher.ts`: версия ключа в самом шифртексте — keyring
поддерживает несколько версий для ротации). `time_unknown` поддержан. Создание профиля
**гейтится** действующим согласием `pd_processing` текущей версии документа (находка
[legal-versioning]) и **сразу считает натальную карту** (`charts/build-chart-input.ts` —
чистый маппинг профиль+пресет→`ChartInput`, покрыт unit-тестами; `charts/
natal-chart-service.ts` — оркестрация `computeChart` + сохранение, `charts.kind='natal'`).

### 3.4 Геокодинг (`packages/shared/src/ports/geocoder.ts`, `apps/api/src/geocoding/`)
- `FixtureGeocoder` расширен методом `suggest()` (список подсказок для AutoComplete).
- `NominatimGeocoder` — реальный адаптер: identifying `User-Agent`, client-side rate-limit
  (1 req/sec по умолчанию), **coord→IANA tz офлайн через `tz-lookup`** (находка [geocoding] —
  Nominatim tz не отдаёт).
- `CachedGeocoder` (apps/api) — обёртка с кэшем `geocode_cache` (запрос→лучший результат),
  соблюдает usage policy без похода в БД на каждый идентичный запрос.

### 3.5 152-ФЗ (`consents-repository.ts`, `routes/consents.ts`, `routes/account.ts`)
- `consents`: отдельная таблица/эндпоинты, версия документа и момент фиксируются; версии —
  единый источник правды `LEGAL_DOC_VERSIONS` (`packages/shared/src/legal/doc-versions.ts`).
- Экспорт (`GET /account/export`) и удаление (`DELETE /account`, требует пароль) — жёсткое
  стирание ПД (email анонимизирован, password_hash — случайный мусор, все refresh-семьи
  отозваны), `status='deleted'`, аудит-запись. Анонимизация UGC — задокументированный no-op-хук
  (UGC-таблиц ещё нет до Ф7, находка [forward-reference]).
- Тексты на русском: согласие на обработку ПД, согласие на маркетинг, политика обработки ПД,
  оферта, дисклеймер — все с плейсхолдерами `{{ЗАПОЛНИТ ЗАКАЗЧИК}}` и явной пометкой
  «черновик, не имеет юридической силы» на страницах.

### 3.6 calc_presets (`packages/db/src/seed/system-calc-presets.ts`, `tools/seed-calc-presets.ts`)
Два системных пресета (doc 20 «Границы MVP» M0/M5): **«Современная западная»** (дефолт;
Плацидус, тропический, орбисы = `DEFAULT_ORB_BY_ASPECT`/`DEFAULT_ORB_BY_BODY` из
`@stassist/astro-core` — единый источник правды с самим ядром) и **«Классическая»**
(целознаковые дома, только мажорные аспекты, без Хирона/Селены — обоснование см. в коде,
опирается на `docs/research/01` §«whole sign — эллинистическое возрождение»). Оффлайн
data-step: `pnpm data:seed-calc-presets` → committed `drizzle/seed/0001_system_calc_presets.sql`
(идемпотентно, `ON CONFLICT (code)`); применение к БД — `pnpm db:seed`.

### 3.7 SPA-кабинет (`apps/web/pages/*`, `apps/web/lib/api-client.ts`)
Новые страницы: `/register`, `/login`, `/verify-email`, `/reset-password` (запрос+подтверждение
в одной странице), `/consent` (152-ФЗ, отдельный экран с чекбоксом), `/profiles` (список +
**empty-state** «У вас пока нет профилей рождения — создайте первый…» + блок аккаунта
экспорт/удаление), `/profiles/new` (форма: AntD `AutoComplete` → `/api/v1/geocode/suggest`,
вывод `tz_id`, выбор calc-пресета, `time_unknown`), `/profiles/@id` (просмотр натальной карты).
`lib/api-client.ts` — fetch-обёртка: Bearer-токен из памяти/`sessionStorage`, CSRF-заголовок на
state-changing запросах, один retry через `/auth/refresh` на 401.

Существующие страницы Ф0 (`/`, `/app`) и их тесты (unit + e2e `hydration.e2e.test.ts`) —
**не тронуты**, проверено вживую заново (2/2 PASS).

## 4. Существенные находки в процессе реализации

1. **`tsx` некорректно резолвит `astronomy-engine` (dual CJS/ESM) при импорте через
   tsconfig-`paths`-редирект из чужого пакета.** `apps/api` теперь по-настоящему импортирует
   `computeChart` из `@stassist/astro-core` (для расчёта карты при создании профиля) — и
   `pnpm --filter @stassist/api dev`/`start` (оба используют `tsx`) в этом случае падают с
   `Cannot read properties of undefined (reading 'Mercury')`. Экспериментально подтверждено:
   ошибка воспроизводится ТОЛЬКО через `tsx`-редирект (напрямую через `node` — компилированный
   `dist`, тот же путь что и прод — работает корректно; через `vitest`/`vite-tsconfig-paths` —
   тоже работает корректно, все integration-тесты и e2e это подтверждают). Причина — вне
   контроля кода приложения (особенность tsx-резолвера). **Обход, подтверждённый вживую**:
   `pnpm --filter @stassist/shared --filter @stassist/astro-core --filter @stassist/db build`,
   затем `node dist/index.js` (`start:prod`) вместо `dev`/`start` при локальной работе с
   auth-функциями, трогающими расчёт карты. `apps/worker` не задет (не импортирует astro-core).
   Это чисто dev-ergonomics ограничение, НЕ влияет на прод (Docker всегда использует
   `node dist/index.js`) и НЕ входит в обязательный гейт.
2. **Барrel `@stassist/shared` в клиентском бандле.** Импорт `PD_PROCESSING_CONSENT_TEXT_RU`
   через `from '@stassist/shared'` тянул в клиентский Vite-бандл серверные порты (nodemailer,
   `node:crypto`) → `vite build` падал («randomBytes is not exported by
   __vite-browser-external»). Исправлено: страницы `/consent`, `/privacy-policy`, `/offer`,
   `/disclaimer` импортируют напрямую `@stassist/shared/legal/doc-versions.js` (глубокий путь,
   без побочных импортов).
3. **`.gitignore` `/_work/tmp/` (с ведущим слэшем)** ловил только корневой `_work/tmp`, но
   стаб-порты (`ConsoleMailer`) пишут туда ОТНОСИТЕЛЬНО `cwd` процесса — при
   `pnpm --filter @stassist/api start` это `apps/api/_work/tmp`. Исправлено на `**/_work/tmp/`.
4. **AntD-специфика в Playwright**: `<Button href>` рендерится как `<a>` (role=link, не
   role=button); `<AutoComplete>` без `name` в `Form.Item` не связывается с `<label>` через
   `for`, а его placeholder — не HTML-атрибут, а оверлей-span (`getByPlaceholder` не находит).
   Обе особенности задокументированы в самом e2e-тесте.

## 5. Integration/e2e — статус

Все написаны **и реально прогнаны** в этой сессии (см. §2), не только «написаны, но
пропущены». Для воспроизведения без этой сессии нужно:

| Пункт | Требуется |
|---|---|
| `pnpm -r test:integration` | `DATABASE_URL` — Postgres 15+ с расширением `vector` (см. `drizzle/migrations/0000_extensions.sql`), миграции применены (`pnpm db:migrate`) |
| `birth-profiles.integration.test.ts` (расчёт карты) | + сид системных пресетов (`pnpm data:seed-calc-presets && pnpm db:seed`) |
| `apps/web/e2e/cabinet-flow.e2e.test.ts` | `RUN_E2E=1`, chromium (`pnpm exec playwright install chromium`), **отдельно запущенный `apps/api`** на `API_PORT` (playwright поднимает только `web`) — см. заголовок файла теста; `MAILER=stub`, `GEOCODER=stub` |
| Реальный SMTP (`MAILER=smtp`, `SmtpMailer`) | Не поднимался (нет MailHog/реального SMTP в среде) — адаптер написан и юнит-проверен на конструирование, `send()` не вызывался вживую |
| Реальный Nominatim (`GEOCODER=nominatim`) | Не поднимался (внешняя сеть к публичному Nominatim не дёргалась) — адаптер юнит-тестирован с моком `fetch` (User-Agent, throttle, парсинг, `tz-lookup`) |

## 6. Плейсхолдеры `{{ЗАПОЛНИТ ЗАКАЗЧИК}}`

- `.env.example`: `NOMINATIM_USER_AGENT` (контакт, было в Ф0), `YOOKASSA_*`, `CADDY_DOMAIN`,
  `STAGE_HOST`/`STAGE_DOMAIN` (Ф0, не тронуты).
- `packages/shared/src/legal/doc-versions.ts`: реквизиты оператора ПД (наименование, ИНН,
  ОГРН, адрес, контакт), факт подачи уведомления в РКН — во всех пяти юр-текстах (согласие ПД,
  согласие на маркетинг, политика, оферта — реквизиты; дисклеймер плейсхолдеров не требует).
  Все страницы (`/consent`, `/privacy-policy`, `/offer`, `/disclaimer`) явно помечены баннером
  «Черновик — не имеет юридической силы» (§8 конвенций: не уходит в прод случайно).

## 7. Известные ограничения / сознательно не сделано

- `pnpm --filter @stassist/api dev`/`start` требуют pre-build workspace-зависимостей при работе
  с функциями, считающими карту (см. §4 находка 1) — задокументировано, не блокирует прод.
- SMTP- и Nominatim-адаптеры не проверялись против реальной внешней инфраструктуры (только
  юнит с моками) — по объективным причинам среды (нет исходящей сети/почтового сервера).
- Пользовательские (не системные) calc_presets — модель данных и API готовы (`calc_presets.
  user_id`), но создание собственного пресета пользователем не входит в объём Ф2 (не
  запрашивалось).
- Публичные страницы `/privacy-policy`, `/offer`, `/disclaimer` — минимальный структурный
  каркас (список пунктов, не итоговый юридический текст); полноценная публичная витрина
  сайта (лендинг, FAQ, «О нас») — Ф3/Ф8 по карте владения контентом (конвенции §6).

## 8. Источники

  `packages/astro-core` (единый источник, не выдумано).
