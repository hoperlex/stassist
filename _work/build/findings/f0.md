# Находки верификации для F0

> Учитывай при реализации. Сквозные системные правила — в `docs/roadmap/31-конвенции-реализации.md` (приоритетны при конфликте с промтом).

## БЛОКЕРЫ (3)
- **[тестовая-стратегия]** Ни один промт не задаёт разделение unit / integration / e2e и env-гейт; заявленные «ворота фазы» (в т.ч. `pnpm -r test`) с первой же фазы тянут pg-boss, readyz-live и миграции и падают без Postgres/docker.
  - _fix:_ Ввести обязательное деление на скрипты: `test:unit` (без I/O, без сети/БД — всегда гоняется в CI и локально) и `test:integration`/`test:e2e` (запуск только при наличии `DATABASE_URL`/`RUN_INTEGRATION=1`, иначе авто-skip). Переопределить «ворота собираемости» каждой фазы = typecheck + lint + test:unit + build (tsc не требует БД); интеграцию/e2e гейтить наличием инфры. Продублировать правило в 30-план-реализации §«Верификация фаз».
- **[worker-boot]** worker на pg-boss подключается к PostgreSQL при старте — процесс не поднимается/висит без БД, ломая smoke, cron-тесты и CI-запуск.
  - _fix:_ Явно предписать ленивую инициализацию очереди: worker импортируется/типизируется/собирается без БД; очередь и cron стартуют только при заданном `DATABASE_URL`, иначе процесс логирует «infra not configured» и не крашится. Тест `ping` и cron-прогон — интеграционные (гейт). В Ф5 cron не должен регистрироваться в unit/build.
- **[docker-в-CI]** Шаги Ф0 «docker-образы api/web/worker (multi-stage)» и «деплой compose+caddy» невыполнимы без docker и как написано входят в CI-гейт.
  - _fix:_ Разнести pipeline: гейт собираемости (typecheck → lint → test:unit → build) не зависит от docker; сборка образов и деплой — отдельные job-ы, запускаемые только там, где docker есть. Локальный dev без docker: поддержать работу против внешнего DSN (облачный Supabase-стейдж по ADR-8) или уже поднятого PG; `docker compose` пометить как опциональный путь, а не единственный.

## MAJOR (7)
- **[db-migration-dependency]** Первая миграция создаёт audit_log с FK actor_id → users, но таблица users появляется только в Ф2 — миграция не применится (relation "users" does not exist).
  - _fix:_ Явно указать: в Ф0 audit_log.actor_id создаётся как `uuid nullable БЕЗ внешнего ключа` (users ещё нет); FK на users добавляется отдельной миграцией в Ф2. Заодно уточнить набор колонок и соглашение id/created_at (updated_at для append-only лога не обязателен).
- **[infra-docker]** docker compose с «PostgreSQL 15» не содержит расширения vector — первая миграция (CREATE EXTENSION vector) упадёт локально.
  - _fix:_ Указать в docker compose образ с pgvector, например `pgvector/pgvector:pg15` (или сборку с установленным pgvector), и явно отметить, что стандартный postgres:15 не подходит для расширения vector.
- **[logging-security]** Список pino-redaction не согласован с проверкой и синтаксически невалиден: 'authorization' отсутствует (а верификация тестирует именно заголовок Authorization), 'birth*' — неподдерживаемый fast-redact синтаксис, указаны имена полей вместо путей.
  - _fix:_ Переписать redaction как список путей и добавить authorization/cookie: например `req.headers.authorization`, `req.headers.cookie`, `req.body.password`, `req.body.token`, `*.email`, и перечислить конкретные поля рождения из док.22 §2 (birthDate/birthTime/placeName/lat/lon) или обернуть их в объект и маскировать `*.birth`. Согласовать формулировку верификации именно с этими путями.
- **[reverse-proxy-verification]** Caddy проксирует только `/api` → api, а health-эндпоинты (/healthz, /readyz) живут в корне api — через публичный вход они попадут на web (404), и проверка `curl /healthz → 200` не пройдёт.
  - _fix:_ Явно добавить в caddy маршруты `/healthz`, `/readyz` (и при необходимости `/metrics`) → api, ЛИБО смонтировать health под `/api/healthz` и поправить URL в верификации; указать, против чего именно (публичный домен или порт api) выполняются curl-проверки.
- **[deploy-target]** «Деплой на стейдж» не имеет входных данных: не задан хост стейджа, домен для TLS caddy, реестр образов и механизм доставки — агент в чистом контексте не сможет ни задеплоить, ни выполнить стейдж-верификацию.
  - _fix:_ Задать конкретику или плейсхолдеры-переменные окружения: STAGE_HOST/STAGE_DOMAIN, реестр (напр. ghcr.io) и метод деплоя (SSH-runner/self-hosted). Для caddy без домена явно разрешить fallback `tls internal` (самоподписанный) в dev/стейдже. Уточнить, что часть верификации (autocannon/SIGTERM/curl) допускается локально, если стейдж-инфраструктура вне зоны ответственности агента.
- **[drizzle-миграции]** Не зафиксировано, что миграции генерируются оффлайн из TS-схемы без подключения к БД; риск, что реализация упрётся в `drizzle-kit` с introspection/pull, требующий живого Postgres.
  - _fix:_ Явно предписать: SQL генерируется из TS-схемы командой `drizzle-kit generate` БЕЗ подключения и коммитится в `drizzle/`; `migrate`/apply (в т.ч. первая миграция с `pgcrypto`/`vector`/`pg_trgm`) — отдельный шаг, гейт по наличию БД. Запретить `drizzle-kit pull`/introspection в build/CI. CI-проверку «схема ⇄ миграции согласованы» делать через generate-diff без БД.
- **[config-fail-fast]** Правило «процесс падает на старте при неполном env» (zod-конфиг) конфликтует с «ключи/DSN добавят позже вручную» — без них api/worker не стартуют, а значит healthz/graceful/autocannon-проверки Ф0 локально не гоняются.
  - _fix:_ Разделить конфиг на переменные, обязательные для СБОРКИ/юнитов, и переменные, обязательные только для рантайма-с-инфрой. Дать `.env.example`/`.env.test` с фиктивными валидными значениями (test-DSN, dummy-ключи), чтобы zod проходил и api поднимался для healthz/graceful-тестов без реальных секретов. `GET /healthz` (liveness) сделать независимым от БД и секретов; 503 без БД ожидается только на `/readyz`.

## MINOR (2)
- **[ci]** Выбор CI (GitHub Actions vs Gitea Actions) отдан «по инфраструктуре заказчика», которую агент в чистом контексте не знает — придётся угадывать.
  - _fix:_ Зафиксировать дефолт (GitHub Actions, т.к. репозиторий на GitHub) с готовым workflow, и одной фразой отметить, что для Gitea Actions синтаксис совместим и меняется только раннер/секреты.
- **[verification-method]** Проверка гидратации «/app гидрируется» не имеет метода: curl её не проверяет, а Playwright-E2E (которого требует общий шлюз фаз из плана) в промте не описан.
  - _fix:_ Добавить минимальный браузерный E2E (Playwright): загрузить `/app`, дождаться гидратации и проверить работу AntD-компонента (клик/состояние). Либо явно прописать ручной чек в браузере с шагами.
