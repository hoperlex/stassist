## 1. Монорепозиторий

```
stassist/
  apps/
    api/                 # Fastify: REST API кабинета, биллинг, LLM-эндпоинты, вебхуки
    web/                 # React 18 + AntD5: SSR-контур (vike) + SPA-кабинет
    worker/              # фоновые задачи: генерация контента, PDF, эмбеддинги, рассылки
  packages/
    astro-core/          # расчётное ядро (чистый TS, без I/O) — §3
    numerology-core/     # матрица судьбы, психоматрица, числа — детерминированные функции
    llm/                 # слой абстракции LLM-провайдеров + промт-конвейер — §4
    shared/              # zod-схемы (единые для API и фронта), типы, константы
    ui/                  # общие React-компоненты (SVG-карта, виджеты)
  tools/                 # скрипты: генерация эфемеридных таблиц Хирона, golden-фикстуры
  drizzle/               # SQL-first миграции
  docs/                  # этот пакет документов
```

pnpm workspaces + turborepo; TypeScript strict; ESLint/Prettier; vitest. CI: типы → линт →
тесты (включая golden-тесты ядра) → сборка → деплой по тегу.

## 2. Контуры исполнения

| Контур | Технология | Задача |
|---|---|---|
| `web` SSR | vike (vite SSR) как middleware **внутри Fastify** | публичные SEO-страницы; AntD5 SSR через `@ant-design/cssinjs` (extractStyle) |
| `web` SPA | те же React-бандлы, hydration | кабинет, чат, мастера заказов |
| `api` | Fastify (отдельный процесс) | REST `/api/v1/*`, auth, биллинг, LLM |
| `worker` | Node-процесс, очередь **pg-boss** (очередь в PostgreSQL — без внешнего брокера) | ежедневная генерация гороскопов, PDF, эмбеддинги, email/push, вебхук-ретраи |

Одна нода на старте (docker compose: web, api, worker, caddy/nginx), горизонтальное
масштабирование — stateless-процессы за балансировщиком; кэш публичных страниц — §6.

## 3. Расчётное ядро `packages/astro-core` (только open-source)

Решение зафиксировано с заказчиком: без платных лицензий. Swiss Ephemeris исключён
(AGPL-3.0 с 2.10.1; серверное использование старых GPL-версий — правовая серая зона).

### 3.1 Состав

| Слой | Реализация | Источник алгоритмов |

| Хирон | предрассчитанная таблица из **JPL Horizons** (1900–2100), чебышёвская интерполяция | `tools/gen-chiron.ts` |

### 3.2 Контракты

- Ядро — чистые функции без I/O: `computeChart(input: ChartInput): ChartData`;
  `ChartInput`/`ChartData` — zod-схемы из `packages/shared` (единственный источник правды,
  используется API, LLM-слоем и фронтом).
- `ChartData` включает блок `meta`: версия ядра, использованные модели/точность, система домов,
  айанамша — попадает в каждый ИИ-отчёт («как считали»).

### 3.3 Гарантии точности (golden-тесты)

- Фикстуры: ≥30 эталонных карт (разные эпохи 1900–2050, широты включая >60°, «время
  неизвестно»), собранные вручную сверкой с astro.com и Sotis; допуски: планеты ≤1′,
  куспиды ≤2′ (у Плацидуса — итеративный допуск), моменты фаз/ингрессов ≤2 мин, соляр ≤1 мин.
- Любое изменение ядра проходит golden-тесты в CI; допуски и известные расхождения
  документируются в `packages/astro-core/ACCURACY.md` (публичная честность о точности).
- Свойства-инварианты (property-based): непрерывность долгот, сумма домов = 360°,
  ретроградность ⇔ отрицательная скорость.

## 4. ИИ-конвейер `packages/llm`

```
ChartInput → astro-core → ChartData (zod) → сериализатор (компактный текст, только факты)
→ ретривер RAG (pgvector: фрагменты интерпретаций по ключам планета×знак×дом, аспектам, арканам)
→ сборка промта (системные правила + данные + фрагменты + задача)
→ адаптер провайдера → LLM → пост-обработка → отчёт
```

1. **Слой абстракции провайдеров**: интерфейс `LlmProvider.generate(req): Result`;
   адаптеры: Anthropic (Claude — приоритетный), OpenRouter, YandexGPT, GigaChat. Выбор и
   fallback-цепочка — конфигурация через env; ключи — секреты (§8). Логи вызовов без ПД.
2. **Анти-галлюцинационные правила** (системный промт): «позиции планет только из блока
   ДАННЫЕ; запрещено вычислять или предполагать положения; при нехватке данных — скажи, чего
   не хватает»; сериализатор передаёт готовые вычисленные факты (вкл. аспекты с орбисами) —
   LLM не делает арифметики.
3. **RAG**: корпус = машиночитаемые фрагменты вики (M7): `interpretation_chunks`
   (ключ: тип×объект×контекст, текст, эмбеддинг). Эмбеддинги — провайдер-абстракция;
   pgvector, HNSW-индекс. Ретрив по ключам детерминированный (точное соответствие
   позиций карты) + семантический (вопрос пользователя в чате).
4. **Пост-обработка**: zod-валидация структуры ответа; фильтры запрещённого (медицинские
   диагнозы, смерть, гарантии событий — regexp+LLM-классификатор); автодобавление дисклеймера;
   сохранение в `ai_reports` с полным аудитом (промт-версия, провайдер, токены).
5. **Стоимость**: кэш по ключу (профиль+тип разбора+версия промта) — повторный запрос не
   платит; лимиты free-тарифа; ежедневные гороскопы генерируются один раз на всех (worker).
6. **Оценка качества**: golden-набор из 20 карт с эталонными разборами; изменения промтов
   прогоняются через LLM-judge + выборочную ручную панель; в проде — кнопка жалобы (M4).

## 5. Backend (Fastify) — требования ТЗ

- **Валидация**: zod-схемы на каждый роут (fastify-type-provider-zod); ответы тоже типизированы.
- **Логи**: pino JSON; redaction-листы: пароли, токены, данные рождения, e-mail (маскирование);
  request-id (correlation id) в каждом логе и ответе.
- **Безопасность**: @fastify/helmet (secure headers), CORS-allowlist (домены портала),
  CSRF-токены для cookie-эндпоинтов (SSR-формы), rate-limit (@fastify/rate-limit: общий +
  жёсткий на auth и LLM-эндпоинты), защита от enumeration в auth.
- **Auth (standalone, по ТЗ)**: access JWT (короткий TTL 10–15 мин, RS256/EdDSA) + opaque
  refresh (httpOnly cookie; в БД — только hash); **ротация refresh при каждом обновлении;
  reuse detection** (повторное использование погашенного токена → отзыв всей семьи сессий);
  пароли — argon2id; password reset одноразовыми токенами (hash в БД, TTL); e-mail-верификация.
- **Жизненный цикл**: health (`/healthz` liveness, `/readyz` readiness с проверкой БД),
  graceful shutdown (drain), пул соединений с лимитами (pg Pool, отдельные пулы api/worker).
- **Файлы**: S3-совместимое хранилище (Yandex Object Storage / MinIO для dev): PDF-отчёты,
  OG-картинки, аватары; выдача через подписанные URL.

## 6. Frontend и кэширование

- React 18 + AntD5 + vike SSR (см. §2); дизайн-токены AntD (кастомная тема), тёмная тема.
- SVG-рендер карты — собственный компонент `packages/ui/ChartWheel` (изоморфный: SSR-страницы,
  кабинет, генерация PNG для шеринга через resvg в worker).
- Кэш публичных страниц: таблица `page_cache` (HTML+TTL) либо заголовки для CDN;
  программатика пересчитывается worker'ом по расписанию (гороскопы — ежедневно в 00:10 MSK).
- Доступность: семантическая разметка, поддержка prefers-reduced-motion; Core Web Vitals
  как CI-бюджет (Lighthouse).

## 7. БД и 152-ФЗ

- PostgreSQL 15+; Drizzle ORM, **SQL-first миграции** (drizzle-kit generate → ревью SQL → apply).
- Расширения: pgvector (RAG), pg_trgm (поиск), pg-boss (очередь).
- **Локализация ПД (решение)**: продовая БД с ПД россиян — **в РФ с первого публичного
  запуска** (Yandex Managed PostgreSQL); с 01.07.2025 первичный сбор ПД россиян в зарубежные
  БД прямо запрещён (ч. 5 ст. 18 152-ФЗ) — вариант «сначала облачный Supabase» юридически
  закрыт. Облачный Supabase — только dev/стейдж с синтетическими данными. Код не зависит от Supabase-специфики (без Supabase Auth/RLS-магии —
  auth свой, §5), поэтому «миграция» = смена DSN + перенос схемы миграциями.
- Данные рождения: шифрование на уровне приложения (AES-GCM, ключ в секретах) для полей
  точного времени/места; согласие на обработку ПД — отдельный документ, фиксация версии
  и времени согласия в БД; экспорт/удаление по запросу субъекта (право на забвение).
- Бэкапы: управляемые снапшоты + логическая репликация; RPO ≤ 24 ч на MVP.

## 8. Секреты и конфигурация

По ТЗ: секреты не в git/образах/фронте/логах/БД. Env-переменные через protected environment
variables / Docker secrets; локально — `.env` вне git + `.env.example`; схема конфига — zod
(`packages/shared/config`), приложение падает на старте при неполном конфиге; pino-redaction
для случайных попаданий секретов в логи.

## 9. Наблюдаемость и эксплуатация

- Метрики: prom-client (`/metrics`) — RPS, латентность, ошибки, длина очередей, LLM-токены/стоимость.
- Sentry (или self-hosted GlitchTip) для ошибок фронта/бэка.
- Алерты: доля 5xx, отставание worker-очереди, провал ежедневной генерации гороскопов
  (страницы «протухают» — прямой удар по SEO), рост стоимости LLM.
- Аудит-лог значимых действий (auth-события, платежи, изменения контента).

## 10. Решения и альтернативы (ADR-сводка)

| # | Решение | Альтернатива | Почему так |
|---|---|---|---|
| 1 | astronomy-engine + собственные дома | Swiss Ephemeris | запрет платных лицензий; AGPL неприемлем для закрытого SaaS |
| 2 | vike SSR внутри Fastify | Next.js | остаёмся в обязательном стеке (Fastify — единственный сервер), меньше сущностей |
| 3 | pg-boss (очередь в PostgreSQL) | Redis+BullMQ | меньше инфраструктуры на MVP; миграция на Redis возможна позже |
| 4 | pgvector в основной БД | внешняя векторная БД | объём корпуса мал (десятки тысяч фрагментов), транзакционность с контентом |
| 5 | Свой auth (JWT+refresh) | Supabase Auth | требование ТЗ (standalone auth) + независимость от Supabase для миграции в РФ |
| 6 | Прод-БД в РФ с запуска | «Supabase, потом мигрируем» | 152-ФЗ: локализация ПД; миграция «потом» = юридический риск с первого дня |
| 7 | pg `page_cache`/CDN | Redis-кэш | детерминированная программатика генерируется заранее, горячий кэш не нужен на MVP |
| 8 | Роль Supabase из ТЗ («на первом этапе») сведена к dev/стейдж-опции | Supabase как прод-БД первого этапа | с 01.07.2025 первичный сбор ПД россиян в зарубежные БД запрещён (152-ФЗ) — облачный Supabase в проде юридически невозможен; код Supabase-agnostic (свой auth, чистый PostgreSQL), подключение через DSN, поэтому Supabase остаётся допустимой dev/стейдж-средой с синтетикой |
