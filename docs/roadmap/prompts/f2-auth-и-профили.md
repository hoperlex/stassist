# Промт Ф2 — Auth, профили рождения, 152-ФЗ

Фаза Ф2 портала «Зодиакум». Прочитай: `docs/architecture/21-техническая-архитектура.md` §5, §7,

`docs/research/05-рынок-услуги-юридика.md` §4 и «Поправки верификации» (152-ФЗ).

## Задача
Standalone-аутентификация по ТЗ + профили рождения с юридическим каркасом персональных данных.

## Требования
1. **Миграции**: таблицы `users`, `refresh_tokens`, `password_resets`, `consents`,
   `birth_profiles`, `calc_presets`, `charts` (по док. 22, поля точно по спецификации).
2. **Auth**:
   - регистрация e-mail+пароль (argon2id), верификация e-mail (одноразовый токен, hash в БД);
   - вход → access JWT (EdDSA, TTL 15 мин, без ПД в клеймах) + opaque refresh в httpOnly
     secure cookie; в БД — только hash refresh;
   - **ротация**: каждый refresh-обмен выпускает новый токен той же `family_id`, старый гасится;
   - **reuse detection**: предъявление погашенного токена → отзыв всей семьи + событие в audit_log;
   - password reset (одноразовый токен, TTL 1 ч, инвалидация сессий по выбору пользователя);
   - rate-limit на auth-эндпоинты (жёсткий), задержки против enumeration (единый ответ
     «если адрес существует — письмо отправлено»);
   - CSRF-токены для cookie-эндпоинтов (SSR-формы).
3. **152-ФЗ каркас**:
   - согласие на обработку ПД — **отдельный экран с отдельным чекбоксом и документом**
     (не в оферте!); версия документа и момент согласия — в `consents`;
   - шифрование полей 🔒 `birth_profiles` (AES-256-GCM, ключ из секретов, ротация ключа
     предусмотрена версией в шифртексте);
   - эндпоинты субъекта ПД: экспорт своих данных (JSON), удаление аккаунта (жёсткое удаление
     ПД + анонимизация UGC-ссылок);
   - страницы: политика обработки ПД, согласие, оферта, дисклеймер (тексты — заглушки для
     юриста, структура готова).
4. **Профили рождения**: CRUD (несколько профилей на пользователя); ввод места — геокодинг
   (Nominatim/OSM с кэшированием в БД + фоллбэк ручного ввода координат) → lat/lon/tz;
   `time_unknown` поддержан; расчёт и сохранение натальной карты (`charts`, kind='natal')
   через astro-core при создании профиля.
5. **Кабинет (SPA)**: страницы регистрации/входа/восстановления, список профилей, форма
   профиля с автодополнением места (AntD AutoComplete), выбор расчётного пресета
   (системные пресеты из док. 20 M0).

## Верификация
- Интеграционные тесты auth: полный цикл; reuse detection (повтор старого refresh → 401 и
  отзыв семьи — проверить в БД); истечение access; CSRF-отказ без токена.
- E2E (Playwright): регистрация → подтверждение e-mail (MailHog) → согласие ПД → создание
  профиля с геокодингом → карта сохранена (проверить `charts.data.meta`).
- Проверка шифрования: значения 🔒-полей в БД нечитаемы (`psql` выборка), экспорт данных
  возвращает расшифрованное.
- pino-логи запросов auth не содержат пароль/токены/дату рождения (grep по логам под нагрузкой).
- Удаление аккаунта: ПД стёрты, вход невозможен, аудит-запись создана.
