# Промт Ф0 — Инфраструктура и каркас

## Контекст
Ты реализуешь фазу Ф0 портала «Stassist» (русскоязычный астрологический сервис).
Прочитай перед началом: `docs/architecture/21-техническая-архитектура.md` (обязательно),
`docs/architecture/22-модель-данных.md` §1 (audit_log), `docs/roadmap/30-план-реализации.md`.
Язык кода-комментариев, коммитов и UI — русский.

## Задача
Создай монорепозиторий и работающие каркасы трёх процессов (api, web, worker) с CI и
деплоем на стейдж. Никакой бизнес-логики — только «скелет, который ходит».

## Требования
1. **Монорепо**: pnpm workspaces + turborepo; структура из §1 док. 21 (`apps/api`, `apps/web`,
   `apps/worker`, `packages/{astro-core,numerology-core,llm,shared,ui}` — пакеты пока пустые
   с index.ts и тестом-плейсхолдером). TypeScript strict во всех пакетах, ESLint+Prettier, vitest.
2. **apps/api** (Fastify): плагины helmet, cors (allowlist из env), rate-limit, request-id
   (генерация + прокидывание в pino), pino JSON с redaction-листом (`password`, `token`,
   `birth*`, `email` — маскирование), zod type-provider; `GET /healthz` (liveness) и
   `GET /readyz` (ping БД); graceful shutdown (SIGTERM → drain → close pool); конфиг через
   zod-схему в `packages/shared/config` — процесс падает при неполном env.
3. **apps/web**: vike SSR как middleware Fastify (второй процесс, свой порт); React 18 +
   AntD5 с SSR-извлечением стилей (`@ant-design/cssinjs` extractStyle); одна страница `/`
   («Stassist — скоро») и одна SPA-страница `/app` с AntD-компонентом; hydration работает.
4. **apps/worker**: pg-boss поверх PostgreSQL; тестовая задача `ping` по расписанию
   (каждые 5 минут пишет лог); подключение к той же БД.
5. **БД**: drizzle-kit настроен, SQL-first миграции в `drizzle/`; первая миграция —
   расширения (`pgcrypto`, `vector`, `pg_trgm`) и таблица `audit_log` из док. 22.
   Локально — docker compose с PostgreSQL 15 + MinIO (S3-совместимое хранилище).
   Код обязан быть Supabase-agnostic (чистый PostgreSQL, подключение по DSN): по ADR-8
   (док. 21 §10) облачный Supabase допустим как альтернативная dev/стейдж-БД с синтетикой —
   поддержи это отдельным DSN в env; прод — только БД в РФ.
6. **Секреты**: `.env.example` со всеми переменными; `.env` в .gitignore; README-раздел
   «Секреты» (env/Docker secrets, запрет секретов в git/образах/фронте/логах/БД).
7. **CI** (GitHub Actions или Gitea Actions — по инфраструктуре заказчика): typecheck → lint →
   test → build; docker-образы api/web/worker (multi-stage, non-root user).
8. **Деплой на стейдж**: docker compose + caddy (TLS, реверс-прокси `/` → web, `/api` → api).

## Верификация (обязательна)
- `pnpm -r test && pnpm -r build` зелёные; CI зелёный.
- `curl /healthz` → 200; `curl /readyz` при остановленной БД → 503.
- Отправь SIGTERM api-процессу под нагрузкой (autocannon) — нет оборванных ответов.
- Страница `/` рендерится сервером (в curl-ответе есть контент без JS), `/app` гидрируется.
- В логах нет секретов: прогони запрос с заголовком Authorization и проверь redaction.
- pg-boss: задача ping видна в логах worker дважды за 10 минут.
