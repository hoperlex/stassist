# Промт: многоагентная редактура и наполнение текстового корпуса «Зодиакум»

## Роль
Ты — главный редактор-оркестратор русскоязычного астрологического портала «Зодиакум».
Задача: довести **текстовый контент** портала с чернового (шаблонного, `quality='draft'`)
до **редакторского продового** уровня — многоагентно, с проверкой. Работай в монорепо
`/root/ClaudeProjects/postoev/stassist`, ветка от `build/stassist-mvp` (создай новую, напр.
`content/editorial`). Всё — на **русском**. Git-коммит по завершении логических блоков; **`.env`
не трогать, секреты не выводить**.

## ОБЪЁМ РАБОТ (scope) — три направления
### A. Рерайт существующего корпуса (основной объём)
Переписать draft-тексты в **уникальные, содержательные, редакторские** — по каждому ключу.
Требования: живой русский язык, конкретика вместо воды, **антидубляж** (тексты соседних ключей
не должны быть перефразом одного шаблона), длина под тип (обзор знака 1500–2500 знаков; планета×знак
600–1200; аспект 500–1000; аркан-позиция 300–700; число 400–900 — ориентир, не догма). Пометить
`quality='reviewed'`.

### B. Генерация недостающего (после аудита покрытия)
Сначала **аудит покрытия**: пройти всех потребителей (страницы `apps/web/pages/**`, сборку
отчётов/PDF, ретрив RAG) и найти ключи/тексты, которых нет или которые тонкие. Заполнить ТОЛЬКО
реальные пробелы (не плодить неиспользуемые ключи). Кандидаты: SEO-мета (title/description)
программатических страниц; осмысленные **empty-states**; копирайт **квиза-онбординга** и **пейвола**
(вопросы, A/B-варианты, маппинг ответ→текст — согласовать с `experiments`/`quiz_answers`);
блок **«как считали»** (человеческое пояснение calc_block); расширение **глоссария** и **FAQ**;
ретро-трактовки/достоинства планет, если их потребляют, но нет. Маркетинг-страницы
(`o-nas`, `methodology`, `tarify`, `redakciya`) — отредактировать до продового тона и E-E-A-T.

### C. Прокачка генераторов рантайм-контента (НЕ пишем руками)
Гороскопы и ИИ-разборы генерируются на лету — их качество даёт **промт/тон**, а не ручной текст.
Улучшить: `horoscope/{tone.ts,build-prompt.ts,phrase-banks.ts,templates.ts}` (вариативность,
конкретика, привязка к астрособытиям, антидубляж зачинов), `prompt/system-rules.ts` и сборку
промта разборов (структура, «как считали», дисклеймер), при необходимости — `forbidden-filters.ts`
(учесть отрицание, не рубить добросовестный текст). Прогон реальным LLM — задача заказчика/рантайма;
ты улучшаешь промты и **тесты** на них.

### НЕ трогать
- Юр-тексты `offer/privacy-policy/disclaimer` и плейсхолдеры `{{ЗАПОЛНИТ ЗАКАЗЧИК}}` (это юрист).
- Фактические данные `celebrities` (даты рождения) — не редактировать как «текст».
- Расчётные движки `astro-core`/`numerology-core` (это математика, не текст).

## Правила качества (для всех текстов — жёстко)
1. **Анти-галлюцинации**: трактовка — да; выдуманные факты/даты/цифры/цитаты — нет. Астрология
   подаётся как **интерпретация**, не как предсказание фактов.
2. **Не-фатализм**: «склонность/тенденция», а не «приговор»; свобода выбора; без «вас ждёт X».
3. **Запрещено** (см. `forbidden-filters.ts`): медицинские диагнозы/лечение, тема смерти,
   гарантии событий/денег, юридические/финансовые «советы как факт». Дисклеймер
   «информационно-развлекательный характер» — уместно и по требованию ТЗ.
4. **Уникальность**: без самоповторов и шаблонных зачинов; каждый ключ — свой смысл.
5. **SEO**: заголовки/мета естественные, ключевые слова без спама; перелинковка по док.23.
6. **Тон**: поддерживающий, конкретный, уважительный; без эзотерического «тумана» и без токсичности.

## Многоагентная структура (как оркестрировать)
Используй воркфлоу/множество агентов (это большой объём — ~1900+ текстовых единиц):
- **Fan-out по категориям и батчам**: раздели ключи по типам (planet_in_sign, aspect, arcanum,
  numerology, wiki-signs, compat-пары, …) и на суб-батчи (~30–50 ключей). Каждый агент-редактор
  пишет свой батч по правилам качества, зная **соседние ключи своей категории** (для антидубляжа).
- **Редактор-верификатор** (отдельный проход на каждый батч): проверяет тон/анти-галлюцинации/
  запрещённое/уникальность/длину; заворачивает и требует переписать плохие.
- **Дедуп-критик** по категории: ловит перефразы-клоны между ключами.
- **Loop-until-done**: повторять, пока не покрыты все ключи и не устранены дубли/пустоты.
- Для блока C — агенты-промт-инженеры + тесты, а не генерация тысяч текстов.

## Формат вывода (контракт — чтобы не сломать портал)
- **Сохрани схему ключей** (`packages/llm/src/facts/keys.ts`) — потребители находят контент по `key`.
- Результат — **обновлённые идемпотентные seed-SQL** тех же файлов (0003/0004/0007/0008) с новым
  текстом и `quality='reviewed'` (upsert по `key`, не ломающий FK/связывание `0009`). Либо
  LLM-генератор, пишущий те же сиды. `pnpm db:seed` должен применяться без ошибок.
- Тексты — валидно экранированы для SQL. Никаких пустых `text`.
- Блок C — правки `.ts` + юнит-тесты.

## Верификация (обязательна перед коммитом)
1. Ворота собираемости зелёные: `pnpm -r typecheck && pnpm -r lint && pnpm -r test:unit && pnpm -r build`.
2. Применение сидов к тест-БД без ошибок; проверка: **нет пустых** `text`; **нет дублей** (одинаковый
   текст у разных ключей); все ключи-потребители покрыты.
3. Прогон текстов через `forbidden-filters` — ноль срабатываний (мед/смерть/гарантии).
4. Выборочная ручная панель (напр. 20 текстов на категорию): доля «плохо» <10%.
5. Отчёт `_report/build/редактура-корпуса.md`: что переписано (по категориям, счётчики
   reviewed/draft), что догенерировано (блок B), что улучшено в промтах (блок C), что осталось.

## Замечание про масштаб и ключ LLM
Ручная многоагентная редактура даёт лучшее качество, но объёмна. Альтернатива/дополнение — прогнать
через реальный LLM (Opus) генератор корпуса с богатым пер-ключевым промтом; тогда этот промт и есть
твой «шаблон качества». Выбери путь под свой бюджет, но контракт вывода (ключи/сиды/reviewed/
верификация) — обязателен в любом случае.
