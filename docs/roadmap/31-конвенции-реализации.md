# 31. Конвенции реализации (addendum к промтам фаз)

> Пакет документов портала «Stassist» · 12.07.2026
> **Обязательно к прочтению каждым фазовым агентом вместе с промтом фазы и планом
> [30](30-план-реализации.md).** Этот документ разрешает сквозные противоречия между промтами,
> выявленные верификацией, и задаёт единые правила, при которых сборка идёт **автономно и
> локально** (Node 22 + pnpm 9, без docker, без живых БД/S3/LLM/платёжки — их env добавляются
> позже). Зафиксированные решения заказчика (open-source ядро без Swiss Ephemeris; прод-БД с ПД
> в РФ; свой auth; LLM — слой абстракции провайдеров) **не пересматриваются** — конвенции их
> реализуют, а не отменяют.

## 1. Уровни тестов и «ворота собираемости фазы»

Каждый пакет/приложение делит тесты на три уровня со скриптами:

- **`test:unit`** — без сети, БД, S3 и файловой системы вне `os.tmpdir()`. Чистые функции
  ядра, zod-схемы, редьюсеры, изоморфные компоненты, стаб-адаптеры. Гоняются всегда.
- **`test:integration`** — требуют Postgres (`process.env.DATABASE_URL`). Оборачиваются
  `describe.skipIf(!process.env.DATABASE_URL)` (или `RUN_INTEGRATION=1`) — при отсутствии инфры
  **авто-skip, не падение**.
- **`test:e2e`** — Playwright; гейт `RUN_E2E=1`.

Vitest: два проекта (`unit`, `integration`) через `vitest.workspace.ts`.

**Зелёный гейт фазы для окружения без docker/БД (обязательный, автономно проверяемый):**
```
pnpm -r typecheck && pnpm -r lint && pnpm -r test:unit && pnpm -r build
```
Интеграция/e2e/деплой — **не входят** в обязательный локальный гейт; они гейтятся наличием
инфры и помечаются в отчёте фазы как «требует ручного шага». Раздел «Верификация» каждого
промта трактуется так: пункты, требующие БД/S3/LLM/платёжки/браузера, — это `integration/e2e`,
их локальный автозапуск skip-ается, а критерий готовности фазы = гейт выше + код этих проверок
написан и включается одним env.

## 2. Порты и адаптеры для внешней инфраструктуры

Любая внешняя зависимость — за интерфейсом в `packages/*`, с реальной И стаб-реализацией,
выбор через env. Это делает Ф4–Ф8 собираемыми и unit-тестируемыми без живых ключей.

| Порт | Пакет | Реальная реализация | Стаб (дефолт локально) | env-переключатель |
|---|---|---|---|---|
| `LlmProvider` | `packages/llm` | Anthropic (приоритет), OpenRouter, YandexGPT, GigaChat | `StubLlmProvider` — детерминированный осмысленный текст с пометкой | `LLM_PROVIDER=stub\|anthropic\|…` |
| `EmbeddingProvider` | `packages/llm` | провайдер эмбеддингов | `StubEmbeddingProvider` — детерминированный вектор | `EMBED_PROVIDER=stub\|…` |
| `ObjectStorage` | `packages/shared` | S3 (Yandex OS / MinIO) | `MemoryObjectStorage` (+ запись в `_work/tmp`) | `STORAGE=stub\|s3` |
| `PaymentProvider` | `packages/shared` | ЮKassa (+СБП) | `FakePaymentProvider` | `PAYMENTS=stub\|yookassa` |
| `Mailer` | `packages/shared` | SMTP | `ConsoleMailer` (лог + запись в `_work/tmp/mail`) | `MAILER=stub\|smtp` |
| `Geocoder` | `packages/shared` | Nominatim (с User-Agent, лимиты, кэш `geocode_cache`) | `FixtureGeocoder` (набор известных городов) | `GEOCODER=stub\|nominatim` |

Стаб-провайдеры **обязаны** давать детерминированный, осмысленный (не «Lorem») результат:
портал должен быть непустым и тесты — стабильными даже без ключей.

## 3. Конфиг: fail-fast в проде, degraded-режим локально

`packages/shared/config` (zod) делит переменные:
- **всегда обязательные**: `NODE_ENV`, порты, `APP_URL`, `COOKIE_SECRET`;
- **обязательные для функции**: `DATABASE_URL`, `S3_*`, `LLM_*`, `YOOKASSA_*`, `SMTP_*`.

Правило: при `NODE_ENV=production` отсутствие обязательных-для-функции → **падение на старте**.
В `development`/`test` процесс **поднимается в degraded-режиме**: логирует
`infra not configured: <подсистема>`, соответствующая подсистема не инициализируется, но
`GET /healthz` (liveness) отвечает 200. `GET /readyz` без БД → 503 (это корректно).

**Worker (pg-boss): ленивая инициализация.** Импорт/типизация/сборка — без БД. Очередь и cron
подключаются только при заданном `DATABASE_URL`; иначе worker логирует degraded и не крашится.
Регистрация cron (гороскопы Ф5 и т.п.) — только в live-режиме, не в unit/build.

## 4. Оффлайн data-steps (генерация данных ≠ сборка)

Скрипты, порождающие **данные** (эфемеридные таблицы Хирона/VSOP87; seed-корпус
интерпретаций; датасет камней; стартовый backfill гороскопов; golden-фикстуры), — это
отдельные команды `pnpm data:<name>`, **не часть `build` и не часть CI-гейта**.

- Их выходы (SQL-сиды `drizzle/seed/*.sql`, JSON-фикстуры, TS-таблицы эфемерид) **коммитятся
  в репозиторий** как источник правды; сборка и тесты их только читают.
- Скрипты с сетью (JPL Horizons для Хирона) держат **закоммиченный кэш** результата в
  `tools/data/`; повторный прогон обновляет кэш, но для сборки не требуется.
- Seed-корпус по умолчанию генерируется на `LLM_PROVIDER=stub` (детерминированно, помечено
  `quality='draft'`), чтобы страницы были непустыми оффлайн; с реальным ключом — качественный
  корпус. Разводим «портал не пустой» (всегда) и «вычитанный корпус» (когда есть ключ/редактор).

## 5. Порядок миграций и общие таблицы

Единый каталог `drizzle/` с нумерованными миграциями; **порядок = порядок фаз**. Миграции
генерируются `drizzle-kit generate` из TS-схемы **оффлайн**, без introspection/pull к живой БД.
Схема БД — единый источник правды: [22-модель-данных.md](../architecture/22-модель-данных.md).

Разрешение перекрёстных FK (таблица нужна ранней фазе, «владеет» поздняя):

| Таблица | Создаётся (skeleton) | Наполняется/расширяется | Правило |
|---|---|---|---|
| `users` | **Ф0** (минимальный набор колонок) | Ф2 (ALTER: verification, аватар, роль и т.д.) | чтобы `audit_log.actor_id` FK был валиден с Ф0 |
| `wiki_articles` | **Ф4** (пустая) вместе с `interpretation_chunks` | Ф7 (контент + editor_id) | чтобы FK `interpretation_chunks.source_article_id` был валиден в Ф4 |
| `compat_pages` | **Ф3** (таблица) | **Ф4** (seed 78 текстов) | Ф3 создаёт таблицу+виджет, Ф4 заливает тексты |
| `geocode_cache` | **Ф2** | — | явно в списке миграций Ф2 |

Правило по умолчанию: если ранняя фаза ссылается на таблицу поздней — создаём её skeleton в
ранней миграции ЛИБО объявляем колонку как `uuid` без FK, добавляя constraint миграцией
поздней фазы. Выбранный вариант фиксируется в промтах обеих фаз (см. §7 точечных правок).

## 6. Карта владения seed-контентом (портал не должен быть пустым)

Единый реестр: кто порождает какой контент и когда. Всё — **оффлайн data-step (§4) + стаб-LLM**,
результат коммитится. Ключи `interpretation_chunks.key` — канонические (ниже).

| Контент | Ключи (`key`) / таблица | Владелец | Потребители |
|---|---|---|---|
| Архетипика: знаки(12), планеты(14), дома(12), аспекты(7), стихии/кресты | `sign:aries:overview`, `planet:mars:overview`, `house:7:overview`, `aspect:square:overview` | **Ф4** seed | Ф5 `/znaki-zodiaka`, Ф7 вики |
| Планета×знак, планета×дом, Asc×знак | `planet_in_sign:mars:leo`, `planet_in_house:venus:7`, `asc_in_sign:leo` | Ф4 | Ф4 отчёты, Ф7 240 стр. |
| Кармические точки (узлы/Лилит/Селена/Хирон) ×знак/дом + их аспекты | `point_in_sign:north_node:leo`, `point_in_house:chiron:6` | Ф4 | natal_full «задачи роста» |
| Аспекты между объектами | `aspect:sun:square:moon` | Ф4 | отчёты |
| Арканы 1–22 × **все** позиции матрицы (7 чакр здоровья, 4 предназначения, линии денег/отношений, возрастные периоды) | `arcanum:3:health_throat`, `arcanum:7:purpose_social`, `arcanum:5:money_line` | Ф4 (полный набор, **не** «22×8») | Ф3 `/matrica-sudby`, Ф6 PDF |
| Нумерология: ЧЖП/выражение/душа/личность 1–9,11,22; 9 ячеек психоматрицы + строки/столбцы/диагонали; перс. год/месяц/день 1–9 | `numerology:life_path:7`, `numerology:matrix_cell:health`, `numerology:personal_year:5`, tradition='numerology' | **Ф4** seed (перенесено из Ф6) | Ф3 `/chislo-puti`,`/kvadrat-pifagora`; Ф6 PDF+циклы |
| 78 текстов пар совместимости | `compat_pages.body_md` | **Ф4** seed | Ф3 `/sovmestimost/{a}-i-{b}` |
| Стартовый backfill гороскопов: сегодня + текущая неделя/месяц/год × 12 знаков × 5 тем + восточный годовой + лунные дни | `horoscopes` | **Ф5** initial-run (seed-режим, стаб-LLM) | SSR-программатика |
| Камни: витрина + соответствия (знак/планета/декада/аркан→камень) + минералог. факты | `stones.*` | Ф6 | `/kamni/*` |
| Статика: лендинг услуг/тарифов, «О нас», FAQ, страницы редакции/авторов (E-E-A-T), юр-дисклеймеры | `.md`/контент-модуль | **Ф8** (каркас публичных страниц — Ф3) | весь сайт |
| Empty-states всех разделов (кабинет: нет профилей/отчётов/чата/заказов; лента; календарь до backfill) | тексты в компонентах | **каждая фаза своего раздела** | — |

**Правило непустоты:** любая страница, публикуемая фазой, обязана иметь непустой осмысленный
контент **или** явный осмысленный empty-state. На SSR идут чанки **и `draft`, и `reviewed`**
(не только reviewed) — иначе портал пуст до ручной вычитки. Статус `draft` отражается через
meta/бейдж «черновик, требует редактуры», а не скрытием страницы.

## 7. Golden-тесты расчётного ядра (Ф1) — исполнимый критерий без выдумки эталона

Агент **не переносит вручную** долготы с astro.com и **не выдумывает** эталон. Вместо этого:

1. **Human-verified seed** `packages/astro-core/fixtures/golden/*.json` — небольшой (3–5 карт)
   набор из общеизвестных/закоммиченных значений; формат:
   `{ input, expected: { longitudes: {...}, houses: {...}, angles: {...} }, source, source_date, verified_by }`.
   Он коммитится и служит жёстким допуском (планеты ≤1′, куспиды ≤2′).
2. **Кросс-валидация** astro-core против **независимого open-source эфемеридного источника**
   (`ephemeris`/Moshier-Ephemeris-JS из npm — детерминированно, офлайн после install) на широком
   диапазоне дат (1900–2050, разные широты) с задокументированным допуском. Это исполнимо
   автономно и покрывает весь диапазон.
3. **Свойства-инварианты** (property-based): непрерывность долгот, сумма домов = 360°,
   ретроградность ⇔ отрицательная скорость.

**Ворота Ф1** = (1)+(2)+(3) зелёные в unit. Расширение seed до ≥30 карт сверкой с
astro.com/Sotis — **human-in-the-loop**, помечается в `ACCURACY.md`, автономный гейт не блокирует.
Система координат фиксируется явно: **видимая геоцентрическая эклиптика эпохи даты**
(нутация + аберрация + световое время); один и тот же движок для натального Солнца и цели
соляра (устранить расхождение VSOP87-«только для соляров»).

## 8. Внешние факты и ПД — вводит заказчик, агент не фабрикует

Реквизиты оператора ПД, ИНН/ОГРН, факт подачи уведомления в РКН, финальные тексты
оферты/политики/согласия, список 100 персон `celebrities` с датами рождения — агент
**не выдумывает**. Он создаёт каркас/шаблоны с плейсхолдерами `{{ЗАПОЛНИТ ЗАКАЗЧИК}}` и
помечает пункт «требует ручного шага». Это прямое следствие принципа против галлюцинаций
из корневого CLAUDE.md.

## 9. Что каждая фаза пишет в отчёт

По завершении фаза дописывает `_report/build/<fN>-отчёт.md`:
- зелёный гейт собираемости (§1) — да/нет, вывод команд;
- что реализовано; какие пункты верификации — `integration/e2e` и помечены «требует ручного шага»
  (какой именно env/бинарь нужен: `DATABASE_URL`, `S3_*`, `LLM_*`, chromium, docker);
- ссылки «источник» на seed-данные; список плейсхолдеров `{{ЗАПОЛНИТ ЗАКАЗЧИК}}`.
