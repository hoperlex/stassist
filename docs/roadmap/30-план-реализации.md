## Принципы

1. Фазы последовательны, внутри фазы — параллелизм по модулям. Каждая фаза имеет
   **критерии готовности (DoD)** и **обязательную верификацию** (тесты + ручной прогон).
2. Расчётное ядро — первым делом и с golden-тестами: на нём стоит всё остальное.
3. Ничего «в стол»: каждая фаза заканчивается деплоем на стейдж и демонстрацией.
4. Юридический каркас (152-ФЗ, дисклеймеры) — не «потом», а в фазе Ф2 вместе с auth.

## Что сознательно за пределами MVP

Джйотиш и ба-цзы как продуктовые модули (версия 1.x; расчётные функции ядра — айанамши,
накшатры, даша, календарь ба-цзы — готовятся «про запас» в Ф1, наружу не выводятся),
маркетплейс консультаций (версия 2, задел — статусная модель заказов и роль expert уже в
схеме БД), мобильные приложения (web-first + Telegram-бот), хорар/электив-инструменты профи,
партнёрская витрина камней.

## Сквозные требования ко всем фазам (из ТЗ)

zod-валидация входов; pino JSON + redaction (ПД, токены); rate-limit; secure headers;
CORS-allowlist; CSRF на cookie-эндпоинтах; health/readiness; graceful shutdown; лимиты пула БД;
request-id везде; секреты только в env/Docker secrets; тесты на каждый инкремент; русский язык
интерфейса и текстов; дисклеймер «информационно-развлекательный характер» на всех отчётах.

## Верификация фаз

**Ворота собираемости фазы (автономно, без docker/БД)** — обязательный минимум по
[31 §1](31-конвенции-реализации.md): `pnpm -r typecheck && pnpm -r lint && pnpm -r test:unit && pnpm -r build`.
Пункты «Верификации» промтов, требующие БД/S3/LLM/платёжки/браузера, — это `integration/e2e`:
локально авто-skip (гейт по env), в отчёте фазы помечаются «требует ручного шага».

Каждый промт содержит раздел «Верификация». Общие ворота фазы:
1. CI зелёный (типы, линт, unit, golden для ядра).
2. E2E-прогон ключевого пути фазы (Playwright): описан в промте.
3. Ручная демонстрация на стейдже (чек-лист в промте).
4. Для контента/ИИ — выборочная ручная оценка качества (панель: 10 примеров, доля «плохо» <10%).

| Риск | Фаза | Митигция |
